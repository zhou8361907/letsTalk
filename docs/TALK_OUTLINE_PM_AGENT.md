## 一、基本信息

| 项 | 内容 |
|------|------|
| **听众** | 前后端开发（不是 PM） |

## 二、叙事主线

### 开场钩子（2 分钟）

> 「你们多少人收到过这样的需求——『在这个页面加个批量审批功能』，然后你去问 PM：
> - 批量上限多少？
> - 失败了一部分怎么处理？
> - 审批谁审？
> - 数据范围怎么隔离？
> PM 回的永远是『你看着办』或者『先做着』。」
>
> 结果就是：写完了，不对，改。改完了，不对，再改。**反复返工。**

**关键点**：这不是 PM 的问题，是**语言差**——PM 说业务语言，开发需要技术语言，中间缺一个翻译。

### 核心论点（3 分钟）

> 「这就是今天要讲的：**一个专门帮开发减少需求返工的 AI 助手。**
>
> 它不是帮 PM 写文档的——是**帮 PM 和开发对齐**的。它做三件事：
>
> 1. **追问模糊点**：PM 说「批量」，它自动追问「上限多少？失败回滚还是继续？」
> 2. **读代码验证**：PM 说「现在流程是这样」，它去读 Controller 确认
> 3. **产出结构化成物**：一份清单，PM 看得懂业务部分，开发直接看代码路径」

### 演示环节（10 分钟）

准备一个真实的演示场景（建议选你们项目里最近一个改过的功能，比如之前的「零金额剔除」）：

**演示流程：**

```
Step 1: 输入一条模糊需求
  「支付计划那边，子表有时候会推一些金额为 0 的数据下来，要把这些数据过滤掉，
   不然影响了后面的流程。」
  → 展示：Agent 没有直接写需求，而是先读代码、追问细节

Step 2: 看 Agent 的反应
  → 展示：Agent 读 Controller、问清「P12/TOTAL_ORIGINAL_COUNT 怎么减」
  → 强调：**这些追问是代替你去问的**

Step 3: 看最终产物
  → 展示：结构化的需求清单（page/control/asIs/toBe/acceptance/codePaths）
  → 强调：**codePaths 里已经有具体的文件路径和 API 信息**
```

### 关键卖点（3 分钟）

用直接打在屏幕上的对比方式：

| 没有助手 | 有助手 |
|---------|--------|
| PM 口头说→你猜→做错→改 | PM 说→助手追问→确认→产出一份结构化成物 |
| 「批量审批做一下」 | 「批量上限多少？失败策略？审批链？」 |
| 你看代码确定现状 | 助手自己读代码写进 asIs |
| 你回忆之前有没有类似功能 | 助手读 hints 告诉你「smbe 有类似的」 |
| 需求文档跟代码对不上 | 助手产出跟代码走的，out of sync 风险低 |

### 他们也试试（2 分钟）

告诉他们怎么自己跑：

```bash
# 1. 拉代码
git clone ...
cd letsTalk

# 2. 装依赖
pnpm install

# 3. 配 API Key（deepseek）
cp .env.example .env
# 编辑 .env，填上你的 LLM_API_KEY

# 4. 启动
pnpm dev

# 5. 浏览器打开，进入 prd 模式，开始写需求
```

### 收尾（2 分钟）

> 「AI 不能替代开发，也不能替代 PM。但它是一个很好的**翻译层**——把业务语言翻译成技术路径，把模糊需求翻译成可落地的规格。
>
> 以前你花 30% 的时间写代码，70% 的时间搞清楚要写什么。
> 这个助手的价值不是帮你写代码——是让你**搞清楚要写什么的时间从 70% 降到 30%**。」

## 三、可能需要准备的

### 演示环境检查清单

- [ ] 提前配好环境变量（LLM_API_KEY）
- [ ] 准备一个 PM 视角的需求文本（复制粘贴即可的）
- [ ] 确保 state.db 有数据（或跑过 import 脚本）
- [ ] 网络通畅（调 API）
- [ ] 大屏幕投影测试

### 常见 QA 预演

| 问题 | 回答方向 |
|------|----------|
| 会不会把我的需求发给别人？ | 不会，API Key 是你自己的，调用 deepseek 的模型，数据不出你的请求 |
| 它写的需求准不准？ | 基于读你本地代码写的，不是猜的。但建议当作**初稿**，审核后再用 |
| 能不能用来写代码？ | 目前聚焦需求整理。后续可以考虑 |
| 要额外学什么？ | 不需要。装好就能用，在浏览器里打字聊天就行 |
| 现有的需求流程要改吗？ | 不用。它输出的是最小公约清单，跟现有 PRD 文档并行不冲突 |
