# AI Requirement OS — V1 页面分析接口与样例

这份文档给当前 V1 提供一份可以直接拿去评审的材料。

当前 V1 的定位已经升级为：

**不是秒级草稿接口，而是 AI Requirement OS 的 Business Map 入口。**

也就是说：

- 它允许花费 1-3 分钟做更深的侦察
- 它优先产出高置信度的前后端关联树
- 它要给后续 Agent 提供可执行的代码坐标

---

## 1. 正式接口

当前建议将下面这个接口作为 V1 正式入口：

```text
POST /api/page-lineage?page_path=绝对页面路径
```

兼容保留旧接口：

```text
POST /api/llm/page-lineage?page_path=绝对页面路径
```

说明：

- 新接口名更贴近当前产品目标
- 输出并不只是 LLM 原始文本，而是结构化 JSON + Markdown 报告
- `lineage` 才是主产物，Markdown 只是展示层
- Agent 只负责输出 JSON，Markdown 由 Python renderer 确定性生成
- 即使没有配置模型，也会返回基于 Evidence Bundle 的 fallback 结果

调试接口：

```text
POST /api/page-lineage/evidence?page_path=绝对页面路径
```

该接口返回方法级裁剪后的 Evidence Bundle，便于排查上下文是否过肥、指针是否命中。

---

## 2. 请求体

```json
{
  "project_name": "RunningAccount-master",
  "frontend_path": "/abs/path/to/vue",
  "backend_path": "/abs/path/to/server",
  "entry_pages": []
}
```

查询参数：

```text
page_path=/abs/path/to/src/views/Detail.vue
refresh=false
```

---

## 3. 响应结构

```json
{
  "mode": "agent",
  "model": "deepseek-chat",
  "warnings": [],
  "lineage": {
    "page_info": {},
    "search_fields": [],
    "display_fields": [],
    "initial_fetches": [],
    "actions": [],
    "backend_overview": {},
    "notes": [],
    "unknowns": [],
    "deep_logic_flags": {}
  },
  "markdown": "# 页面数据解析报告..."
}
```

字段说明：

- `mode`
  - `agent`：已调用模型，基于 Evidence Bundle 直接输出 JSON
  - `fallback`：未调用模型，使用 Evidence Bundle 推导出的保底结果
- `lineage`
  - V1 的主产物
- `markdown`
  - 从 `lineage` 派生出的可读报告
- `actions[].request_flow[].backend_pointers`
  - 后续 Agent 读取真实后端代码的精准坐标
- `deep_logic_flags`
  - 当逻辑过深或不适合继续总结时，明确打标而不是幻觉补全

## 6. 这份样例适合别人点评什么

建议让别人重点看这几件事：

1. 这个 JSON 结构够不够清晰
2. Markdown 是否足够产品可读
3. `initial_fetches` 和 `actions` 的粒度是否合适
4. 当前是否还需要继续精简，或者增加一两个必要字段
5. 哪些信息应该留给后续 Agent 工具搜索，而不是塞进首版报告

---

## 7. 当前样例的已知限制

这份样例是有意做轻的，当前已知限制包括：

- `payload_fields` 还没有自动补出
- `backend_entry / services / tables` 还比较浅
- `return_summary` 仍偏占位
- `修改 / 下一步` 这类操作的业务文案还可以再润

但它已经足够作为一版 V1 讨论样品。
