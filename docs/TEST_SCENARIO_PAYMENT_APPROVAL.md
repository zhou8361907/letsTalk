# 测试场景：支付计划批量审批

## 设计目标

渐进式揭示复杂度的需求场景，测试 Agent 的：
1. **Persona 转变** — 是否像资深 PM 一样挑战模糊假设
2. **深度探询** — 是否自动识别模糊词/隐式假设
3. **Hints 利用** — 是否主动 `read` 相关 hints
4. **桥梁能力** — 产出对 PM 和研发都有用

## 场景背景

SMIFC 系统已有手工支付计划功能。各省医保局每月生成支付计划→人工审批→推银行支付。
现在四川医保局想加「批量审批」功能，后续宁夏也想跟。

## 对话流程（测试用）

### 第 1 轮 — PM 提出初始需求

> "我们四川医保局那边觉得现在一个个点支付计划太慢了，想做个批量审批功能。就是勾选多条支付计划，然后一次提交审批。宁夏那边说后面也想要这个功能。"

**期望 Agent 行为：**
- ✅ Persona 转变：以一概要或追问开头，而不是「我来协助你整理需求」
- ✅ 识别「批量」→ 追问上限、失败策略
- ✅ 识别「审批」→ 追问审批链、角色
- ✅ 识别「四川」「宁夏」→ 区分首批 vs 后续、差异点
- ✅ 读 hints（module-overview、banking-domain、regional-differences）
- ❌ 不直接猜规则写成定论 toBe

### 第 2 轮（如果 Agent 追问了，PM 回答）

PM 回答：
> "上限的话……目前一个地市一个月大概 50-200 条吧，全省可能上千条。失败了肯定不能全部回滚啊，哪条没过提示哪条就行。审批的话还是走现在那套——科室主任审完→局领导审，两级。四川和宁夏流程一样的，就是宁夏的银行参数不一样。"

**期望 Agent 行为：**
- ✅ 记录上限、部分成功策略、审批链
- ✅ 注意「银行参数不一样」→ 追问具体差异
- ✅ asIs 写清楚现状（已有单条审批流程）
- ✅ codePaths 包含现有 Controller/页面路径
- toBe 标注待确认项

### 第 3 轮（继续深入）

PM 补充：
> "对了，批量审批通过之后，支付计划要自动推银行支付，不要再人工点一次了。现在单条审批通过后是人工点「支付」才推银行的。"

**期望 Agent 行为：**
- ✅ 识别「自动」→ 追问触发条件
- ✅ 识别跨模块影响（审批→支付执行→银行通信）
- ✅ 读 banking-domain hint 了解支付流水线
- ✅ 考虑 smsbm 模块影响
- ✅ 考虑宁夏 smip 覆盖差异

### 第 4 轮（边界情况）

PM 说：
> "宁夏那边说他们银行用的农行，跟四川的工行不一样。还有之前宁夏有一些定制的审批流程，不知道这次能不能统一。"

**期望 Agent 行为：**
- ✅ 注意 smbe 模块银行适配器差异
- ✅ 考虑 smip 区域定制覆盖
- ✅ 在 openQuestions 中标注「统一 vs 分离」
- ✅ 产出包含研发参考视图

## 预期最终产物

产出应包含：
- **PM 视图**：用业务语言描述 page/control/asIs/toBe/acceptance
- **研发参考**（codePaths 中的结构化内容）：
  - API 合约示意
  - 涉及模块列表（smfr/smps + smsbm + smbe）
  - 省份差异点（四川工行 vs 宁夏农行）
  - 错误处理模式（部分成功策略）
- toBe 中标注待确认项
- openQuestions 分类标注
