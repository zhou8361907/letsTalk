# 测试场景 2：银行支付通道切换与对账调整（跨省）

## 设计目标

与前一个场景互补，侧重测试：
1. **hints 触发** — Agent 是否主动调 `get_business_hints` → `read banking-domain.md`
2. **跨模块感知** — smsbm（银行通信）+ smbe（银行适配器）+ smrm（对账）
3. **区域差异处理** — 不同省份的银行配置不同
4. **隐式假设** — 切换期间的过渡方案、回滚策略

## 场景背景

四川原来用工行（ICBC HTTP REST）做医保支付，省局要求切换到农行（ABC WebService）。宁夏也是同样的情况，但宁夏原来用的就是农行，**不需要切换，只需要调整对账频率**。

这个场景涉及：
- 四川：银行切换（ICBC → ABC，协议从 HTTP 到 WebService）
- 宁夏：对账调整（已有 ABC，只需改对账批次配置）
- smsbm 模块：银行通信层
- smbe 模块：ABC 适配器是否已有
- smrm 模块：对账配置

## 对话流程

### 第 1 轮 — 抛出需求

> "省局要求把四川的医保支付银行从工行换成农行。宁夏那边说他们本来就是农行不用换，但是对账频率想从每天一次改成每天两次。技术那边说工行是 HTTP 的，农行是 WebService 的，协议不一样可能有点麻烦。"

**期望 Agent 行为：**
- ✅ 调 `get_business_hints` → 看到 `banking-domain.md`、`regional-differences.md`
- ✅ `read banking-domain.md` → 了解 ICBC HTTP vs ABC WebService 差异、EncryptEnum
- ✅ `read regional-differences.md` → 了解四川和宁夏的 smip 覆盖
- ✅ 识别四川（切换）和宁夏（调整）是两件不同的事，不该混为一条需求
- ✅ 追问：工行存量数据怎么处理？切换窗口期？
- ❌ 不直接猜 ABC 适配器是否存在

### 第 2 轮 — PM 补充细节

> "切换的话，四川这边新数据走农行，工行那边已经发出去的支付继续跟踪，不用切回来。ABC 的适配器之前宁夏接入的时候好像写过，但不确定四川能不能直接用。存量数据的话，工行那边在途的走完就不发了。宁夏对账频率那个改了就能生效的吧？"

**期望 Agent 行为：**
- ✅ 读代码验证 ABC 适配器是否已有（smbe/abc/ → 确认存在）
- ✅ 识别「四川能不能直接用」→ 查 smip 下四川是否有定制覆盖
- ✅ 区分「适配器复用」vs「配置差异」（银行 IP、证书、CA 各省不同）
- ✅ 追问：对账频率调整是改配置还是改代码？

### 第 3 轮 — 深入

> "代码看了应该能复用，宁夏的 ABC 适配器是通用的，就是两边的银行 IP、端口、证书那些不一样。对账频率那个改配置就行，有个批次步长的参数。另外省局要求切换期间每天出一个迁移报告，对比工行停用前和农行启用后的支付成功率。"

**期望 Agent 行为：**
- ✅ 理解「适配器复用 + 配置不同」模式（各省 bank.properties）
- ✅ 在需求中区分：四川需求 1（银行切换）、宁夏需求 1（对账调整）、增量需求 2（切换期监控报告）
- ✅ 识别「迁移报告」→ 追问具体指标、展示方式、持续多久
- ✅ 产出包含配置路径（bank.properties 位置、对账步长配置项）

## 预期最终产物

应产出 **3 条需求条目**：

**条目 1（四川）：支付银行从工行切换为农行**
- page: 支付管理 → 银行配置
- asIs: 四川医保支付走 ICBC HTTP REST，smbe/icbc/ 适配器在用
- toBe: 切换为 ABC WebService，复用 smbe/abc/ 适配器，配置四川专属 bank.properties（IP/端口/证书/CA）
- codePaths: smsbm（通信层）、smbe/abc/（适配器）、smip/sichuan/（区域定制）

**条目 2（宁夏）：对账频率调整**
- page: 对账管理
- asIs: 对账每天一次，批次步长配置为 1 天
- toBe: 每天两次，调整批次步长参数
- codePaths: smrm 对账模块，批次配置项

**条目 3（四川 + 切换期）：迁移监控报告**
- page: 支付管理 → 迁移监控
- toBe: 切换期间日报，对比工行停用前/农行启用后的支付成功率、失败原因分布
- codePaths: smsbm 对账/统计相关
