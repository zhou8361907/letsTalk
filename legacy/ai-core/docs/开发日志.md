# AI Requirement OS — 开发日志

本文档用于记录这个项目**当前已经做了什么、接下来准备做什么、哪些能力暂缓**。  
目标是始终守住节奏，避免再次把重心过早压到沙箱和自动改码上。

---

## 1. 当前共识

当前阶段我们已经明确把项目拆成三步走：

1. **V1：页面级扫描与说明书**
   先把“看懂一个页面”做好，稳定输出页面级前后端关联信息。
2. **V2：页面问答 Agent**
   在 V1 的基础上，让 Agent 具备工具调用、短期记忆、后台扫描和多轮问答能力。
3. **V3：代码修改与沙箱闭环**
   最后再做代码计划、diff、应用、回滚，以及更成熟的沙箱演示能力。

一句话总结：**先把页面分析做深，再把 Agent 做强，最后再把沙箱做真。**

## 3. V1 正在定义的核心产物

### 3.1 PageDataLineage JSON

当前 V1 的核心，是围绕页面生成一份结构化 JSON。  
这份 JSON 是后续 Markdown 报告、问答 Agent、甚至沙箱数据底座的统一来源。

当前约定的主结构：

```json
{
  "page_info": {},
  "search_fields": [],
  "display_fields": [],
  "initial_fetches": [],
  "actions": [],
  "backend_overview": {},
  "notes": [],
  "unknowns": []
}
```

其中重点是：

- `search_fields`：查询条件字段
- `display_fields`：表格/表单展示字段
- `initial_fetches`：页面初始化请求
- `actions`：按钮或关键操作对应的接口链路

### 3.2 Markdown 报告

Markdown 报告不会再单独自由发挥，而是：

```text
PageDataLineage JSON
    ↓
Markdown Renderer
    ↓
页面数据解析报告
```

也就是说，**Markdown 只是展示层，JSON 才是事实来源。**

当前补充约束：

- `actions` 只优先保留关键业务操作，不收录过于简单的前端行为
- `display_fields` 当前先只覆盖主表格字段
- `unknowns` 只保留在 JSON 中，Markdown 默认不展示
- 可通过后续工具再次搜索到的细碎实现信息，不强行塞进 V1 主报告

## 5. V2 计划

当 V1 站稳后，再进入页面问答 Agent：

- Agent 读取页面分析 JSON 作为主上下文
- Agent 可以自主调用工具读取代码、搜索代码、追踪接口
- 支持短期记忆
- 支持多轮问答
- 支持后台任务持续扫描或补全页面知识

这个阶段的重点不是“改代码”，而是：

**让 Agent 像一个真的页面分析助手，能和产品或研发围绕页面持续沟通。**

---

## 6. V3 计划

V3 才处理下面这些更重的能力：

- 代码修改计划
- diff 预览
- 应用修改
- 回滚修改
- 更真实的前端沙箱
- 基于真实页面结构的高拟真交互演示

当前这些内容不是放弃，而是**延后**。

---

## 7. 当前明确暂缓的内容

为了保证 MVP 节奏，下面这些内容先不作为当前主线：

- 大规模沙箱交互打磨
- 复杂页面视觉原型能力
- 自动直接修改真实业务代码的完整闭环
- 深度数据库血缘审计
- 大而全的多 Agent 编排
- 复杂审批流、低代码 DSL、微前端适配

---

## 8. 接下来讨论顺序

接下来建议严格按这个顺序推进：

1. 定 `PageDataLineage v1`
2. 定 `Markdown 报告模板 v1`
3. 定 Agent Prompt v1
4. 打通 Agent 生成 JSON / Markdown
5. 再讨论 V2 的问答与记忆

---

## 9. 一句提醒

这个项目后面当然可以长成 Agent + 沙箱 + 代码修改平台，  
但**现在最重要的是先把“页面级理解能力”做成一个稳定产品**。
