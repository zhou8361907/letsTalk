# 第一章 背景与痛点

## 1.1 一个典型的工作日

**上午 10:00，某 PM 小张** 正在写「医保缴费记录查询」的需求文档。这个功能用户一直在催，但小张不确定系统里是否已有类似实现。他打开系统，翻了几十个菜单页面，还是没找到——只能去问对应模块的研发。

**上午 10:30，某研发小李** 正在开发「个人账户收支明细」功能。刚写到一半，被小张打断：「帮我看看缴费记录查询的现有逻辑在哪？」小李停下手中的活，开始 grep 代码、翻 Controller、定位接口……十五分钟后终于找到，把路径发给小张，然后才能继续自己的开发。

**下午 3:00，新来的研发小王** 接手一个修改需求。他需要了解「字典管理」功能的数据流向，但不知道代码在哪、枚举怎么定义的。在群里问了一圈，没人马上回复，他只能自己边翻代码边猜，一个简单的问题花了将近一个小时。

**下午 5:30，需求评审会** 上，小张写好的需求文档被研发打了回来——「这个接口其实已经上线了」「这个逻辑和现有的认证模块冲突」「这个字段定义需要和财务组确认」……需求需要大改。

---

**这不是某个人的问题，这是我们整个部门每天都在重复的故事。**

## 1.2 四个核心痛点

### 痛点一：代码知识传递效率低

- **新人上手慢**：熟悉部门业务代码平均需要 **2-4 周**，期间大量时间花在找人问「这是什么」「在哪找」
- **"活文档"依赖重**：资深开发成为行走的 API 文档，日常被各种「这个接口在哪」「这个逻辑怎么走的」打断
- **巨量代码难定位**：核心业务逻辑散落在 1000-3000 行的 Java Controller 中，人工 grep + 翻页定位效率极低

### 痛点二：PM 写需求犹如「盲人摸象」

- PM 不熟悉代码实现细节，需求描述和实际情况常有偏差
- 经常出现「其实已有该功能但 PM 不知道」「要新增的功能其实已有接口」
- 需求文档与代码实现脱节，研发拿到需求后还需要大量「考古」才能理解真实上下文
- 来回沟通成本高：一个需求从起草到定稿通常需要 3-5 轮确认

### 痛点三：系统菜单与代码路径割裂

- 企业系统拥有 **4000+ 菜单条目**，覆盖多个子系统
- PM 认识「用户管理」「缴费查询」「字典维护」这些业务名称，但不认识对应的 `UserAdmin.vue`、`PaymentQueryController.java`
- 菜单名与代码路径之间没有可靠、自动化的映射关系
- 跨模块调用链路难以追溯

### 痛点四：知识流失与重复劳动

- 系统黑话（"抵扣""收支明细""停用"等）每个新人理解不同
- 过去做过的需求、踩过的坑，换个人就要重来一遍
- Commit Message 信息量不足，无法回溯决策原因
- 文档更新滞后，大部分文档写完后很快就和代码对不上了

## 1.3 为什么是现在？

- **AI Coding Agent 已趋成熟**：Claude Code、Cursor、Copilot 等工具已经在业界证明了「AI 理解代码」的能力
- **国产大模型可用**：DeepSeek 等国产模型在代码理解、中文语境上的表现已经达到实用水平
- **通用工具不够用**：ERP 研发的特殊性（封闭内网、单体巨石仓库、复杂业务黑话、企业级菜单）决定了通用 AI 工具无法直接落地
- **定制化方案正当时**：letsTalk 正是为解决这些"通用工具管不到"的问题而生的

# 第二章 项目介绍

## 2.1 一句话定位

> **letsTalk 是一个"代码库 AI 对话助手"，让 PM 和研发用自然语言提问就能理解代码、梳理需求，彻底从"人找代码"变为"代码找人"。**

## 2.2 核心设计理念

| 理念 | 通俗解释 |
|------|---------|
| **按需理解，而非全量编译** | AI 像查字典一样——每次只读当前问题需要的代码片段，而非预先编译整个仓库。读得少反而理解更准 |
| **Agent 自主探索** | AI 像资深开发者一样自己用 grep/read/find 在代码中穿梭，不需要人类一步步告诉它去查哪里 |
| **代码永远是权威** | 所有回答必须基于真实代码，杜绝 AI 幻觉。找不到就是找不到，不编造 |
| **PM 与研发双视角** | 既满足研发的技术查询需求，也提供 PM 可用的业务理解和需求产出工具 |

## 2.3 技术底座

- **运行时**：基于 Pi SDK Agent 框架 + DeepSeek 大模型，全栈 TypeScript（Node.js）
- **前端**：Next.js 15 + React 19，现代 Web 架构
- **模型**：DeepSeek（可切换其他 LLM，支持信创适配）
- **部署**：单进程内网部署，代码数据不出域
- **仓库规模**：适配大型单体 Java/Vue 仓库（已验证 10 万+ 行代码级别）

## 2.4 当前阶段

| 维度 | 状态 |
|------|------|

| Java AST 方法级提取 | ✅ 已完成 |
| 企业菜单锚点集成 | ✅ 已完成 |

| 对话持久化与恢复 | ✅ 已完成 |
| 多格式导出 | ✅ 已完成 |

# 第三章 核心能力与应用场景

## 3.1 能力一：自然语言代码问答

**一句话**：你问代码答，Agent 自主搜索直到找到答案。

**场景故事**  
研发小李想知道「订单状态的枚举定义在哪」。他打开 letsTalk，输入问题，Agent 自动 grep 整个仓库，找到枚举类、定位字段、附带引用路径——整个过程 **不到 30 秒**。同样的操作，手动 grep 至少需要 3-5 分钟，还不算问人的时间。

PM 小张想知道「系统里有哪些地方用到了字典功能」，Agent 会跨模块搜索所有引用点，从后端 Service 到前端页面，逐一列出，并附上业务描述。

**关键特性**：
- Agent 自主决定搜索策略（grep→read→find），多轮检索直到找到答案
- 无需预先配置索引或向量数据库
- 搜索结果附带代码路径和行号引用，可追溯、可验证

**传统方式 vs letsTalk**：

| 对比项 | 传统方式 | letsTalk |
|--------|---------|----------|
| 代码定位 | 手动 grep / 问人 | 一句话提问 |
| 耗时 | 5-20 分钟 | 10-30 秒 |
| 准确性 | 依赖个人经验 | 基于代码全文匹配 |
| 覆盖范围 | 知道的才能找到 | AI 跨模块搜索 |

## 3.2 能力二：锚点/聚焦模式

**一句话**：锁定一个文件或模块，深度分析——首轮即可回答，无需等待工具调用。

**场景故事**  
研发小王在做 Code Review，需要理解一个 1500 行的 `OrderController.java`。他点选该文件作为锚点，Agent 自动读取文件前 150 行作为"预览"，常见问题（这个 Controller 有哪些接口？用的什么 Service？）在第一轮对话就能直接回答，不需要等 AI 先搜一轮。

**关键特性**：
- 支持锚定 Vue 页面、Java Controller、菜单项、任意文件
- JIT（Just-In-Time）上下文注入：首轮常见问题秒回
- 可在"全库模式"和"聚焦模式"间切换

## 3.3 能力三：需求分析助手

**一句话**：左聊右写，PM 和 Agent 聊需求，右侧自动生成结构化需求清单。

**场景故事**  
PM 小张要写「个人账户收支明细」的需求。他打开 letsTalk 的需求助手模式，左侧是对话区，右侧是需求清单面板。他先选择一个菜单锚点，Agent 自动加载当前页面的代码结构。

「我想加一个按年月筛选的功能」——小张用自然语言描述，Agent 理解后，在右侧自动生成一条需求条目：
- **业务描述**（PM 可读）：支持按年份和月份筛选收支明细
- **代码线索**（研发可用）：前端对应页面的数据请求方法、后端 Service 层查询接口、数据库表结构

Agent 还会主动标注缺项（比如「这个需求的验收标准是什么？」用 🟡 标记），帮 PM 提前想周全。需求清单可导出为正式 PRD 文档。

**传统方式 vs letsTalk**：

| 维度 | 传统方式 | letsTalk 需求助手 |
|------|---------|-------------------|
| 需求来源 | PM 凭经验/感觉 | PM + Agent 基于真实代码 |
| 技术可行性 | 研发评审时才知道 | 写需求时 Agent 已核实 |
| 完整度 | 容易遗漏细节 | Agent 自动标注缺项 |
| 研发理解成本 | 高（需翻译业务描述） | 低（附带代码线索） |
| 导出 | 人工整理 | 一键导出 Markdown |

## 3.4 能力四：Java AST 手术刀式提取

**一句话**：千行级 Java Controller 精确提取，不再"读不全"。

**场景故事**  
面对 2000 行的 `PaymentController.java`，研发小李需要快速找到「缴费记录查询」接口的具体实现。他不关心这个文件的其他 40 个接口，只想看这一个。

letsTalk 先用 `list_methods` 列出所有方法签名（秒级响应），然后 AI 选择性地用 `read_method` 精确提取目标方法的完整代码——包括参数校验、业务逻辑、异常处理、SQL 调用，完整无截断。

**为什么需要这个能力？**  
ERP 系统的 Java Controller 文件普遍在 1000-3000 行，通用 AI 工具的 `read` 调用会触发截断或括号不匹配，导致模型理解偏差。letsTalk 的 AST 工具通过方法签名解析 + 括号平衡算法，实现了"只读需要的，读得完整"。

## 3.5 能力五：企业菜单锚点

**一句话**：PM 认菜单名，AI 自动找对应的代码。

**场景故事**  
PM 小张说「改一下用户管理」。他在 letsTalk 的菜单树中找到「系统管理 > 用户管理」——这是他所熟悉的业务导航。点选后，Agent 自动解析该菜单项对应的代码路径（前端页面 + 后端 Controller + Service），直接进入分析状态。

**关键特性**：
- 对接 MySQL `sys_menu` 表，实时同步 4000+ 菜单条目
- 三级门户级菜单导航，PM 无需学习代码路径
- 菜单名 → 代码路径的自动映射补齐了 PM 和研发之间的"语言鸿沟"

## 3.6 能力六：记忆系统——越用越聪明

**一句话**：跨会话自动积累业务知识，但又不会过时。

**关键特性**：
- **分层记忆**：从用户偏好（L0）、业务黑话索引（L1）、主题知识（L2）到代码状态（L3），层层递进
- **自动提取**：Agent 在对话中遇到新的业务术语时，可以主动记录到记忆库
- **不替代代码**：记忆仅作为参考辅助，回答仍以实时代码为准，自动标注可能的过时风险
- **跨会话复用**：今天学到的东西，明天提问时 Agent 还记得

## 3.7 能力七：多格式导出

**一句话**：一次 AI 对话，沉淀为正式交付物。

- **对话全文导出**：Markdown 格式，完整保留问答记录
- **PRD 需求导出**：仅导出业务层描述，适合 PM 评审
- **研发附录导出**：包含代码线索、技术分析，适合研发参考（AI 懒生成，需要时触发）

# 第四章 技术架构概览

## 4.1 架构示意

```
┌──────────────┐       ┌──────────────────────────────────┐       ┌────────────┐
│  用户浏览器   │ <──> │        letsTalk Server            │ <──> │ DeepSeek   │
│  (Next.js)   │ SSE  │  (Node.js, 单进程内网部署)          │ API  │  大模型    │
└──────────────┘       │  ┌────────────┐ ┌─────────────┐  │       └────────────┘
                       │  │ JIT 上下文  │ │ Pi Agent    │  │
                       │  │ 组装器      │ │ 引擎        │  │
                       │  └────────────┘ └──────┬──────┘  │
                       │         ↕              ↕         │
                       │  ┌──────────────────────────┐    │
                       │  │  自定义工具集              │    │
                       │  │  grep / read / AST / 记忆  │    │
                       │  └──────────┬───────────────┘    │
                       └─────────────┼────────────────────┘
                                     ↕
              ┌──────────────────────┼──────────────────┐
              ↕                      ↕                  ↕
    ┌─────────────────┐  ┌──────────────────┐  ┌──────────────┐
    │ workFront (Vue) │  │ workBack (Java)  │  │ .agent 运行时 │
    │  前端源码        │  │  后端源码         │  │ 对话/记忆/菜单 │
    └─────────────────┘  └──────────────────┘  └──────────────┘
```

## 4.2 核心设计原则

| 原则 | 解释 | 对用户的意义 |
|------|------|------------|
| **单进程部署** | 一个 Node.js 服务搞定一切，无依赖链 | 部署简单，几乎零运维成本 |
| **内网优先** | LLM 请求走内网代理，代码数据不出域 | 满足信息安全合规要求 |
| **按需加载** | 不预先编译仓库，只读取当前需要的代码 | 适用于大型仓库，启动快，资源省 |
| **可插拔模型** | 基于 DeepSeek，可切换其他 LLM | 可适配信创要求，不被单一模型绑定 |
| **Pi 优先** | 尽量复用 Pi Agent 框架的原生能力 | 不自研轮子，稳定且兼容上游升级 |

## 4.3 部署形态

| 阶段 | 部署方式 | 访问方式 |
|------|---------|---------|
| 当前开发 | `pnpm dev` 单机运行 | 绑定本地仓库，localhost 访问 |
| 部门试用 | 内网服务器 Docker 部署 | 团队通过浏览器访问 |
| 未来生产 | 可嵌入企业微信/飞书/内部平台 | 统一入口 |

# 第五章 应用成效与价值分析

## 5.1 直接收益（预估）

| 指标 | 预估提升 | 测算依据 |
|------|---------|---------|
| **代码查找时间** | **减少 60-80%** | 从平均 10-20 分钟（grep + 问人）→ 30 秒 AI 回答 |
| **新人上手周期** | **缩短 40-50%** | 从 2-4 周 → 1-2 周可独立回答代码问题 |
| **PM 需求返工率** | **降低 30-50%** | 需求基于代码核实，减少「已有功能重复提」和「不可实现的需求」 |
| **需求文档完整度** | **提升 50%+** | 结构化模板 + Agent 自动标注缺项，减少遗漏 |
| **跨模块知识传递** | **效率提升 3-5 倍** | Agent 一键跨模块搜素，替代人工逐个询问 |
| **研发被打断次数** | **减少 40-60%** | 减少「这个接口在哪」「这个逻辑怎么走的」类重复提问 |

> **以上为初期保守估算**。实际效果会随工具使用深入和记忆系统积累而持续提升，我们将在试点阶段收集真实数据进行校准。

## 5.2 间接收益（长期影响）

- **减少「活文档」依赖**：资深开发无需反复解答重复性问题，专注更有价值的工作
- **需求-代码对齐**：PM 写的需求天然附带代码索引，研发无需额外"考古"
- **知识资产化**：每一次对话都是知识沉淀，记忆系统持续积累部门业务语境
- **PM 能力成长**：PM 通过 Agent 阅读代码，逐步建立对系统实现的技术 sense，写需求更贴合实际
- **团队协作提效**：跨模块、跨小组的代码问题不再需要"找对人"，问 Agent 即可

## 5.3 成本分析

| 成本项 | 说明 | 预估费用 |
|--------|------|---------|
| **算力** | DeepSeek API 调用，按 token 计费 | 初期每月数百至千元级别 |
| **运维** | 单进程 Node.js 部署 | 几乎零运维成本 |
| **人员维护** | 作为部门内部工具 | 1-2 人兼职维护即可 |
| **硬件** | 内网服务器（已有资源复用） | 无新增硬件成本 |

## 5.4 ROI 总结

> **以一个 50 人研发部门计算，平均每人每天节省 20-30 分钟的代码查找/沟通时间，全年可节约约 4,000-6,000 人时。按人均工时成本估算，年节约价值远高于工具本身的开发与运行成本。**

更关键的是，**这些节约的不仅仅是时间，更是团队的注意力和工作节奏**——研发不再被频繁打断，PM 能写出更靠谱的需求，新人更快上手产生价值。

# 第六章 推广落地路线图

## 6.1 总体策略

```
小范围验证 ──→ 扩大试点 ──→ 全面推广 ──→ 持续深化
  种子期        试点期        推广期        深化期
  1-2 周        2-4 周        1-2 月        持续
```

## 6.2 阶段一：种子期（1-2 周）

**目标**：核心功能走通，收集首批真实反馈

- **动作**：
  - 选定 **1 名 PM + 1-2 名研发** 作为种子用户
  - 绑定真实项目代码仓库
  - 走通 3-5 个真实业务场景（如 PM 写一个需求、研发查一个接口、新人了解一个模块）
  - 收集问题清单和改进建议
- **交付物**：内测反馈报告 + 问题清单
- **成功标准**：种子用户确认工具对其日常工具有帮助

## 6.3 阶段二：试点期（2-4 周）

**目标**：在实际项目环境中验证价值，完成体验打磨

- **动作**：

## 附录 B：名词解释

| 术语 | 通俗解释 |
|------|---------|
| **Agent（智能代理）** | AI 不再只是「你问我答」，而是能自己决定"先查哪、再读哪"的智能体。就像给 AI 装上了手和脚，可以自主操作工具 |
| **AST（抽象语法树）** | 代码的"解剖结构图"。letsTalk 的 AST 工具相当于一把精确的手术刀，能从几千行代码中精准切出某个方法的完整代码 |
| **JIT（即时上下文）** | 需要时才读对应的代码片段，不是一次性把所有代码塞给 AI。效率高、精度好、适用于大型仓库 |
| **SSE（服务器推送事件）** | 一种让服务器能实时推送数据到浏览器的技术。用户提问后，AI 的回答像打字一样逐字出现，体验流畅 |
| **Pi SDK** | letsTalk 底层的 Agent 框架，提供了 AI 自主调用工具、多轮对话、会话管理等核心能力 |
| **grep** | 在代码中进行全文搜索的命令，相当于在整个仓库里"关键词搜索" |
| **提示词（Prompt）** | 给 AI 的指令和上下文配置，决定了 AI 的行为方式和能力边界 |
