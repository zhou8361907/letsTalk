# letsTalk PM 智能体 — 宣讲演讲稿

> 面向部门前后端开发同事 | 建议时长 25 分钟 | 含现场演示

---

## 开场（2 分钟）

**（打开一页写着「需求」两个字的 PPT，或者直接白板写）**

大家好，今天花 20 多分钟讲一个东西——一个最近在做的 AI 助手，叫 **letsTalk**。

讲之前我问一个问题：**你们多少人经历过这种场景？**

PM 走过来跟你说：「支付计划那边，帮我加个批量审批功能。」

然后你开始反问：
- 批量上限多少？
- 失败了一部分怎么处理？
- 审批谁审？
- 数据范围怎么隔离？

PM 回的永远是「你看着办」或者「先做着」。

然后你写完了，不对。改。改完了，不对，再改。**反复返工。**

**（停一下，让这句话落下来）**

这不是 PM 的错，也不是你的错。这是**语言差**——PM 说业务语言，开发需要技术语言，中间缺一个翻译。

letsTalk 就是这个翻译层。

它不是帮 PM 写文档的，是**帮 PM 和开发对齐**的。今天我想让你们看看它到底能做什么，以及——它对你们每天的工作意味着什么。

## 什么是 AI Agent（5 分钟）

**（这节给不太了解 Agent 的同事）**

先花 3 分钟说一下，**什么是 AI Agent**，因为一会儿演示时你们会看到它"自己在那干活"。

你们用过 豆包 吧？你问一句，它答一句——这叫 **Chatbot**。

Agent 不一样。Agent 不是等你问一句答一句，而是：
1. **你给它一个目标**
2. **它自己决定怎么干**——先查什么、再读什么、用什么工具
3. **干完回来告诉你结果**

举个例子，传统的 AI 像这样：

```
你：这个 Controller 有哪些方法？
AI：等一下，我没有代码……
你：（粘代码）现在呢？
AI：哦，有这些方法……
```

Agent 是这样：

```
你：这个 Controller 有哪些方法？
Agent：（自己 grep 找到文件 → 自己调 list_methods → 读结果）
Agent：有 12 个方法，分别是……
```

**区别在哪？** Agent 有**工具**——它能调 grep、调 read、调 find、调 memory。它像一个人一样在代码里穿梭。

**（可以画个图，或者直接投这张）**

```
用户在编辑器中输入 "帮我重构这个函数" 并按 Enter
  │
  ▼
┌──────────────────────────────────────────────────┐
│ 阶段 A：TUI → AgentSession                        │
│ InteractiveMode.run()                             │
│   getUserInput() → "帮我重构这个函数"              │
│   session.prompt(userInput)                       │
└────────────────────┬─────────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────────┐
│ 阶段 B：消息预处理 (AgentSession.prompt)           │
│   1. 检查是否斜杠命令 → 否                         │
│   2. 扩展事件触发                                  │
│   3. 展开 skill / prompt template                  │
│   4. 验证模型和 API key                           │
│   5. 检查是否需要压缩（compaction）                │
│   6. 构建 AgentMessage[]                          │
│   7. _runAgentPrompt(messages)                    │
└────────────────────┬─────────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────────┐
│ 阶段 C：Agent 启动 (Agent.prompt)                 │
│   normalizePromptInput() → AgentMessage[]         │
│   runPromptMessages(messages)                     │
│     → runWithLifecycle() → runAgentLoop()        │
└────────────────────┬─────────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────────┐
│ 阶段 D：★ Agent Loop（核心循环）                    │
│                                                  │
│   while (true) { // 外层：follow-up 循环           │
│     while (hasMoreToolCalls || pendingMessages) { │
│                                                  │
│       ① streamAssistantResponse()                │
│          → LLM 调用，stream 回复                  │
│                                                  │
│       ② 检查 stopReason                          │
│          → error/aborted → 退出                   │
│                                                  │
│       ③ 提取 tool calls                          │
│          → 有 → executeToolCalls()               │
│          → 无 → hasMoreToolCalls = false          │
│     }                                             │
│                                                  │
│     检查 follow-up messages                       │
│     → 有 → 设为 pending → continue 外层循环        │
│     → 无 → break，退出                            │
│   }                                              │
└──────────────────────────────────────────────────┘
```

这叫 **Agent 循环**——思考→行动→观察→再思考，直到完成目标。

翻译成人话就是：

1. LLM 收到你的问题，决定「我要先 grep 一下这个关键词」
2. 它调 grep → 拿到结果 → LLM 说「哦这里有个 Controller，我 read 一下」
3. 调 read → 拿到代码 → LLM 说「行了我知道了」
4. **退出循环**，给你最终答案

**关键在于**：每一步都是 LLM 自己决定的——不是我们写死的逻辑。我们只是给了它一套工具和规则，它自己在那循环直到任务完成。

### 四个核心概念

| 概念 | 什么意思 | 类比 |
|------|---------|------|
| **循环** | LLM 调用 → 出 tool call → 执行工具 → 结果喂回 LLM → 再出 tool call → … → 直到 LLM 说完成了 | 像人写代码：写一行→跑一下→看结果→再改。最内层循环每秒可能跑好几轮 |
| **工具调用** | Agent 能调 grep/read/find/list_methods 等 | 像给人配了一套瑞士军刀 |
| **记忆** | 跨会话记住业务知识（用户偏好、术语、惯例） | 像团队 Wiki，但自动维护 |
| **Skills** | 可复用的行为规范文件 | 像操作手册，告诉 Agent 遇到情况怎么做 |

**现在我们来看这东西实际跑起来什么样。**

## 演示一：老页面加新功能（6 分钟）

**（打开 letsTalk，确保 PRD 模式，准备好复制文本）**

我演示一个真实的场景。这是我们最近的一个需求——**支付计划受理前加确认**。

**（粘贴这段给 Agent）**

> "内蒙古支付计划那边，现在点受理就直接走审批了，有时候点错了也没法撤回。想在受理前面加一道确认，用户勾选数据后点确认按钮，确认成功后才能点受理。支持批量确认。"

**（指着屏幕）** 注意看它接下来的动作——它不是直接开始写需求，而是：

**（让 Agent 跑起来，解说每一步）**

1. **先读 hints** —— 它调了 `get_business_hints`，去看了项目背景知识
2. **读代码** —— 它去读了 `NmPayPlanController`、`PayPlanController`，确认现有受理逻辑
3. **读前端** —— 它去读了支付计划申请页面的 Vue 代码

**（等 Agent 产出第一条清单后，指着屏幕说）**

看，它自己已经把现状摸清楚了：内蒙古有专属的 `NmPayPlanController`，基线 `PayPlanController` 有 `confirmPayPlan` 方法，但那是做别的事的。它把这些全写进了 `asIs`。

**（等 Agent 追问后）**

现在它在追问了——「确认状态存哪里？PayPlan 主表加字段还是扩展现有枚举？」

**（如果现场有人笑，就补一句）**

**这些追问，平时是你去问 PM 的。现在它帮你问了。**

**（等最终产出后）**

最终产物是这样一份东西：

```
[ready] 支付计划增加受理前确认环节
  toBe: 
    1）列表页新增确认状态列
    2）新增确认按钮（支持批量）
    3）未确认不可受理
    4）退回/驳回自动重置
  codePaths: NmPayPlanController.java, PayPlanController.java
```

**这份清单，PM 看得懂业务部分，你直接看 codePaths 就知道改哪。** 以前你花半小时问清楚的东西，它 3 分钟帮你理完了。

## 演示二：新页面新功能（7 分钟）

**（切到另一个对话）**

再看一个更复杂的——**从零做一个新功能**。

**（粘贴给 Agent）**

> "现在追回违规医保基金全靠线下手工，内蒙想做一个线上的追回单功能。从发现违规开始，发起到科室审核、领导审批、推银行扣款、到账确认，全流程线上走。独立的菜单页，不是挂现有页面。"

**注意看——这次是全新功能，系统里没有现有实现。**

**（指屏幕）** 看，它先读了 `pm-new-feature.md`——这是我们给 Agent 写的新功能评估指南。然后它开始做一件事：**查重**。

**（等 Agent 扫完代码后）**

有意思了——它发现新疆、海南已经有「待遇追回」相关的代码，基线还有 `smr_treatment_recover` 表。但它的结论是：「现有能力主要停留在到账对账环节，不是从发现违规发起的全流程追回单」。

**这说明它不是傻傻地重复造轮子，也不是简单地"有就算了"。它做了判断。**

**（等 Agent 追问后）**

现在它在问关键问题了：追回对象是只针对机构还是包含个人？审批层级固定两级还是金额大要往上走？追回单要不要关联基金收入账？

**而且你注意看右上角——它把「工作流统一用 Activiti」这条记到 memory 里了。下次再聊到审批流程，它自动知道这个惯例。**

**（等最终产出后）**

最终产出是两条需求：

```
[ready] 内蒙追回单全流程功能
  MVP 建议：一期先做填写+审批+到账确认，银行扣款接口预留不阻塞
  影响范围：涉及 smr（基金）+ smsbm（银行）+ 审批流

[draft] 工作流统一使用 Activiti + 页面配置
```

**这个例子的关键不是它写了什么，而是它怎么做的：先读指南→查重→复用扫描→MVP 建议→影响范围分析→追问→产出。** 这一套流程，是资深 PM 的做事方式，不是速记员的。

## Agent 是怎么做到的（3 分钟）

**（这节解释原理，给开发同事）**

你们可能好奇，它怎么"自己决定先查什么"的？背后的机制其实不复杂：

**（投这张图）**

```
                    ┌──────────────────────┐
                    │  LLM（大模型大脑）     │
                    │  理解意图 + 决定下一步  │
                    └──────┬───────────────┘
                           │
          ┌────────────────┼────────────────┐
          │                │                │
    ┌─────┴─────┐  ┌──────┴──────┐  ┌──────┴──────┐
    │  工具调用   │  │   记忆系统   │  │  Skills 规则 │
    │ grep/read  │  │ USER/CORE   │  │ AGENTS.md   │
    │ find/AST   │  │ topics/INDEX│  │ hints/*.md  │
    │ memory     │  │ 跨会话记忆   │  │ pm-prd.ts   │
    └────────────┘  └─────────────┘  └─────────────┘
```

**三样东西决定了它的行为：**

### 1. 工具调用

它手里有这些工具：
- `grep` / `find` —— 搜代码
- `read` / `read_method` —— 读文件 / 读 Java 方法
- `list_methods` —— 看 Controller 有哪些接口
- `get_business_hints` —— 看业务提示
- `update_requirement_draft` —— 写需求清单
- `memory` / `save_memory` —— 读写记忆

每轮循环中，LLM 决定「我现在该用什么工具」，调完后看结果，再决定下一步。

### 2. 记忆系统

分三层：
- **Tier 1**（USER / CORE）：你的偏好、团队的惯例——每轮自动注入
- **Tier 2**（topics + INDEX）：业务术语的解释——按需自动匹配
- **Tier 3**（会话历史）：跨会话的上下文——用过 session_search 召回

你们刚才看到的「工作流统一用 Activiti」被记住，就存在 Tier 1 的 CORE 里。

### 3. Skills 和行为规范

我们给它写了几套规则：

| 文件 | 作用 |
|------|------|
| `AGENTS.md` | 通用原则 + PM 四原则（先想再写/最小范围/手术刀修订/验收驱动） |
| `pm-prd.ts` | PRD 模式的具体战术规则（看到模糊词怎么追问、隐式假设怎么探测） |
| `hints/pm-new-feature.md` | 新功能评估指南（合理性/复用性/影响范围） |
| `hints/banking-domain.md` | 银行对接领域知识 |
| `hints/module-overview.md` | 模块速查 |

**这就是为什么刚才演示时，它遇到新功能会自动去读 pm-new-feature.md——不是巧合，是我们写进了行为规范。**

## 对比总结（2 分钟）

**（投这个对比表）**

| 没有 letsTalk | 有 letsTalk |
|---------|--------|
| PM 口头说 → 你猜 → 做错 → 改 | PM 说 → Agent 追问 → 确认 → 产出结构化清单 |
| 「批量审批做一下」 | 「批量上限多少？失败策略？审批链？」 |
| 你花 30 分钟读代码确定现状 | Agent 自己读代码写进 asIs |
| 需求文档跟代码对不上 | Agent 产出跟代码走的 |
| 你回忆之前有没有类似功能 | Agent 扫模块告诉你「新疆海南有类似的」 |
| 新人：这个接口在哪？老人：这个怎么走？ | 问 Agent，30 秒 |

**收尾：**

> AI 不能替代开发，也不能替代 PM。
>
> 但它是一个很好的**翻译层**——把业务语言翻译成技术路径，把模糊需求翻译成可落地的规格。
>
> 以前你花 30% 的时间写代码，70% 的时间搞清楚要写什么。
>
> **这个助手的价值不是帮你写代码——是让你搞清楚要写什么的时间从 70% 降到 30%。**

## QA 预演

**Q：会不会把我的需求发给别人？**
A：不会。API Key 是你自己的，调的是 DeepSeek 的模型，数据只在你请求和模型之间传递。我们不存你的需求数据。

**Q：它写的需求准不准？**
A：基于读你本地代码写的，不是猜的。但建议当作**初稿**，审核后再用——跟任何人写的需求一样需要评审。

**Q：要额外学什么？**
A：不需要。装好就能用，浏览器里打字聊天就行。

**Q：它能写代码吗？**
A：目前聚焦需求整理。后续可以考虑扩展到代码生成。

**Q：现有的需求流程要改吗？**
A：不用。它输出的是最小公约清单，跟现有的 PRD 文档并行，不冲突。
