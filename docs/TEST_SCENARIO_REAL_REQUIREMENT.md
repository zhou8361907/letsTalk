# 测试场景：支付计划零金额剔除与 N001 批次拆分改造

## 需求来源

真实业务需求（用户提供），涉及支付计划模块的**数据接收过滤**与**批次拆分规则变更**。

## 测试目标

1. Agent 是否能理解业务背景中的领域术语（子表/主表/孙表、P12、N001、险种账套）
2. Agent 是否主动读 hints 了解支付计划/银行对接上下文
3. Agent 对**已规范需求**的处理——不瞎猜规则、不制造多余问题、但也不漏关键细节
4. 产出物是否能直接给研发使用

## 对话流程

### 第 1 轮 — 只抛背景，不抛需求

角色扮演 PM：

> "最近省医保局那边提了个问题，我们子表里现在会推一些支付金额为 0 的数据下来，而且这种数据没有关联的孙表数据。另外在批次拆分的时候，如果拆成 N001 批次，里面可能会混着职工、居民、救助这些不同险种的数据，导致业财记账走不通，因为不同险种账套不同。"

**期望 Agent 行为（规范需求的验证）：**
- ✅ Persona：像资深 PM 一样理解业务背景，而非急着填格子
- ✅ 识别核心问题点（零金额数据、险种混排）
- ✅ 记录 context（为后续需求做铺垫）
- ❌ 不要猜规则（PM 还没说解决方案）
- ℹ️ 可以追问背景细节，但不应过度

### 第 2 轮 — 抛需求 1：零金额剔除

PM 接着说：

> "所以第一块需求是：数据接入的时候，如果发现子表金额是 0，就把这条子表数据剔掉，同时主表的 P12（实付笔数）和 TOTAL_ORIGINAL_COUNT（应付笔数）减一，减按实际 0 的条数来扣。有同事建议说接入时加个处理事件，对已接入的数据重新刷新一下这两个字段。"

**期望 Agent 行为：**
- ✅ 准确理解 P12、TOTAL_ORIGINAL_COUNT 是什么
- ✅ 识别「处理事件」→ 这是个模糊表述，追问触发时机/频率
- ✅ asIs 写清楚现状（子表含 0 金额数据可入）
- ✅ toBe 写清楚新行为（接入时过滤 + 计数修正）
- ✅ 记录相关页面 URL
- ℹ️ 可读 hints 了解支付计划数据模型

### 第 3 轮 — 抛需求 2：N001 批次拆分变更

PM 接着说：

> "另外一块是关于 N001 批次的。现在如果子表拆分后发现有负值的情况，原来是不拆了，统一放一个单独的 N001 批次里。现在想改成：有负值的也正常按批次拆分规则走，拆分出来的批次号改成 N001 + 原来批次号 + 拆分标识。还有就是强调一下，所有 N001 开头的批次号，正常走审批流程，但不能点'待遇支付'，只能手工支付——这个本来就是现在这样，保持就行。"

**期望 Agent 行为：**
- ✅ 理解批次拆分规则变化（从「不拆→合并」到「正常拆+前缀标记」）
- ✅ 识别 N001 批次的约束（可审批、不可待遇支付）
- ✅ 区分「新增行为」（拆分规则）vs「强调已有行为」（审批/支付限制）
- ✅ 标注待确认项（如果有任何不清楚的）
- ✅ 产出包含两个相对独立但有关联的需求条目

## 预期最终产物

应包含 **2 条需求条目**（不要合并成 1 条）：

**条目 1：数据接入时零金额子表剔除**
- page: 支付计划初审-受理
- asIs: 子表可含 0 金额数据进入系统，关联主表 P12/TOTAL_ORIGINAL_COUNT 包含这些无效数据
- toBe: 接入时过滤 0 金额子表数据，主表对应计数递减
- codePaths: 涉及数据处理接入的 Controller/Service

**条目 2：N001 批次拆分规则调整**
- page: 支付计划初审-受理
- asIs: 负值子表统一单独放 N001+原批次号，不参与正常拆分
- toBe: 负值子表按正常规则拆分，批次号改为 N001+原批次号+拆分标识
- 强调：N001 批次可审批不可待遇支付（保持现有）
- codePaths: 批次拆分逻辑相关代码
