## 0. 产品定位

### 0.1 一句话

**帮 PM 借助读代码，把口语收成自己认得、研发也能照着做的需求；文档可选。**

### 0.2 是什么 / 不是什么

| 是 | 不是 |
|----|------|
| PM 分析需求时的「读代码 + 收成」助手 | 写码工具（Cursor / Copilot 等仍负责改代码） |
| 流程上的正向补充 | Jira、需求流转平台 |
| 锚点 + 只读工具核实现状 | 无锚点全库扫描 |

### 0.3 三条能力（其它皆衍生）

1. **观察**：有锚点时读代码，用业务话说明「现在怎样」
2. **推演**：改动可能影响什么（锚点范围内，业务语言一句即可）
3. **收成**：右侧清单收拢「改哪、现在、要改、怎么验」；缺口标出，不连问

### 0.7 主 / 次（v0.5 真源原则）

| 层级 | 内容 | 谁定稿 | 何时生成 |
|------|------|--------|----------|
| **主** | PM 与 Agent 确认的业务格（页面/改哪里/现在/要改/验收） | **PM** | 对话过程中 |
| **次** | 研发附录（前后端可能涉及的线索） | **Agent 读代码** | **导出时 lazy**（另调模型），非每轮对话 |

- **定稿、验收、扯皮只认「主」**；研发附录是添头，开发会在仓库里自行核对。
- 附录**不必严谨**；文首声明「非 PM 定稿，请以代码为准」。
- 样例见 `docs/examples/requirement-export-v0.5-sample.md`。

### 0.4 产物（按优先级）

| 优先级 | 产物 | 说明 |
|--------|------|------|
| P0 | 左侧对话 | 短；不单独做「评价」模块或总评区 |
| P0 | 右侧需求清单 | 上：业务格；下：该条的研发参考（`techTraces`） |
| P1 | 菜单选锚点 | 见 §4 |
| P2 | 导出 Markdown | 可选；与清单同构，每条需求下附研发参考 |

**不强制输出文档。** PM 在 letsTalk 里想清楚即可；需要时再导出。

### 0.5 开发阶段原则

- **少就是好**：不加「评价步骤」「八章定稿」「verified 状态机」等产品壳
- **对话能短就短**；右侧格子少而清晰
- **技术分开展示**：业务格在上；`techTraces` 挂在**每条需求下方**（不必折叠隐藏，字号可略小）
- **先达成 PM 侧目标，再拓展**（研发回写、Jira、受击面自动化等）

### 0.6 与编码工具链

```text
PM 口语 → letsTalk（读代码、收清单）→ 可选 Markdown → Cursor 等写码
```

letsTalk 止于「需求说清楚」；不替代下游。

## 1. 背景与痛点

- PM 不懂现有页面/逻辑，也不知道研发需要哪些信息，只能自然语言描述
- 研发从散文里考古，返工多（要加删除其实已有、要表格其实是树等）
- Agent 价值：**以代码为准**，帮 PM 收成高质量需求，而不是替 PM 写八股 PRD

---

## 2. 已采纳的交互原则（历史评审摘要）

| 原则 | 说明 |
|------|------|
| 双轨 | 左对话 + 右清单；清单是 PM 思考的外化 |
| 无锚点不读代码 | 用 hints / 菜单名破冰；有锚点才核对 |
| 最多 1 个阻断问题 | 其余缺口在右侧标 🟡，不问卷 |
| PM 一件事 ≈ 清单 1 条 | 禁止拆「前端一条、后端一条」 |
| 不做整段 PRD 每轮注入 | 清单快照 + 工具更新 |

## 3. 目标形态（v0.4）

### 3.1 界面

```text
┌──────────────┬─────────────────────────┐
│ 对话（短）     │ 需求清单（业务为主）        │
│ PM 口语       │ ① 标题                    │
│ Agent 带过现状 │    现在怎样 / 希望改成 / 验收 │
│ 必要时 1 问   │    🟡 缺哪格               │
│              │    研发参考（该条 techTraces，列表）  │
└──────────────┴─────────────────────────┘
```

- **不**在右侧做「Agent 评价」专区；现状写在「现在怎样」格或对话里一句带过
- **业务格**不写表名、类名、`modify` 等；**研发参考**区可写路径/接口，与业务分开即可

### 3.2 数据结构：业务为体、技术为用

每条需求 **一条记录**，分两层（存储可映射当前 `RequirementItem`，产品语义如下）：

```typescript
// 目标形态（命名可调整，语义固定）
interface RequirementItemV2 {
  id: string;              // 稳定 id，如 req_1；新建时工具传 "new"
  title: string;           // 一句话标题
  business: {
    page?: string;         // 在哪个页面（业务名，非路径）
    control?: string;      // 改哪里
    asIs?: string;         // 现在怎样（Agent 读过代码后写，PM 可认）
    toBe?: string;         // 希望改成
    acceptance?: string;   // 怎么验收
    scope?: string;        // 适用范围，可选
  };
  techTraces: string[];    // Agent grep/read 得到的线索，展示在该条 business 下方
  gaps?: string[];         // 还缺什么（可选，或由空格状态推导）
}
```

| 层 | 谁关心 | 内容 |
|----|--------|------|
| `business` | **PM 主读** | 大白话；每格 1～3 句，不是关键词堆砌 |
| `techTraces` | **该条需求下方** | 文件路径、方法、表、接口等；PM 需要时可扫一眼，研发直接可用 |

**禁止**把技术词写进 `business` 各格；读代码所得一律进 `techTraces`。

### 3.3 工具契约：单条 upsert（非字段 patch）

**第一性原理：** 模型擅长「重写一条完整需求」，不擅长改 JSON 深层字段。

- 工具名（目标）：`upsert_requirement`（可替代现有 `update_requirement_draft`）
- **每次只提交一条**的全量 `business` + 当前 `techTraces`（traces 建议 append 去重，不全量清空除非改口）
- 参数 `id`：`"new"` 或已有 `req_xxx`
- 服务端按 id 覆盖该条；返回当前所有 id 列表
- **每轮 prompt 前**注入清单快照（已有 `requirement_draft_snapshot`，保持）

**禁止：**

- `update_field(id, field, value)` 式微调
- 一次传多条又不带 id 导致 merge 出空壳第二条
- PM 改口后仍保留矛盾旧条（应 upsert 同 id 或显式作废，作废规则待实现）

### 3.4 清单「够用」规则（简单）

必填（业务格）：

- `toBe`、`acceptance`
- 有锚点时：`asIs` 由 Agent 填，PM 未反对即可

可选：`scope`、影响一句（业务语）

右侧用 🟡 标缺格即可；**不**单独做「就绪 / 评价 / 评分」状态机。

### 3.5 导出（v0.5）

| 按钮 | 产物 | 说明 |
|------|------|------|
| **导出 PM 定稿** | `*-PM定稿.md` | 仅业务格 = 真源 |
| **导出含研发附录** | `*-完整含研发附录.md` | 定稿 + 分隔线 + Agent lazy 整理（实验） |
| 探索 / 无清单 | `*-对话.md` | 聊天实录 |

附录生成：导出时单次调用模型 + 只读 grep/read；**不修改**清单字段。

旧模板 `prd-template.md` 等不作正文结构。

## 4. 锚点：菜单地图（P1）

**问题：** PM 认「用户管理」，不认 `UserAdmin.vue`。

**方向：** 把代码树伪装成 **系统菜单**（外部评审「破局点二」，分阶段落地）。

| 阶段 | 做法 |
|------|------|
| P0 | 继续用手选锚点列表 + 手填路径（现状） |
| P1a | 静态 `menu-map.json`：由脚本读路由或菜单表生成 `[{ menuName, path, component }]` |
| P1b | 前端搜索/点选菜单 → `component` 作为 `anchor` 传给 Agent |
| P1c | 工具 `search_menu_by_name`：PM 口语「改退款」时 Agent 查 map 设锚点 |

**注意：** 菜单数据源须与真实系统一致（路由 vs 库表菜单），实现前先核实 workFront。

## 7. 示例（用户管理 · 改性别）

**PM：** 用户管理加性别；删除改成切换性别；保留树。

**Agent（对话，短）：** 这是树形列表，已有「删除此用户」；用户资料里还没有性别字段。切换要确认弹窗吗？（最多这一问）

**右侧一条需求：**

| 格 | 内容 |
|----|------|
| 标题 | 删除改为切换性别并展示性别 |
| 现在怎样 | 树节点只显示用户名；行上有「删除此用户」；无性别 |
| 希望改成 | 展示性别；删除改为男↔女切换；弹窗可选性别 |
| 怎么验收 | （空 → 🟡） |

**研发参考（该条下方）：** UserAdmin.vue、User 实体无 gender、…

**不应出现：** 第二条「后端支持」；清单里写 `rbac_user` / `el-tree`。
