# 新功能场景下的 Agent 思维增强 — 合理性·复用性·影响范围

> 讨论稿 v0.1 | 2025-06-05
> 关联文件：`packages/context/src/prompt/pm-prd.ts`

---

## 1. 为什么需要这个

现有 `PM_PRD_RULES` prompt 在"改现有逻辑"场景表现良好——Agent 能通过 read/grep 读现有代码填写 `asIs` 和 `codePaths`，在"深度探询"中追问模糊表述和隐式假设，最终以 `readyToFinalize` 条件控制出口质量。

但在"新功能"场景有明显缺口。当 `asIs` 为空（代码中找不到已有实现）时，Agent 的典型行为是：

1. 反复读码尝试"找到"不存在的代码，浪费 token
2. `codePaths` 无法填写 -> 违反 `readyToFinalize` 条件 -> 卡死
3. 退化成向 PM 追问技术细节（"银行接口是否支持跨省？""用什么图表组件？"）
4. `openQuestions` 堆积 PM 无法回答的实现层问题，导致对话死锁

**核心问题**：Agent 对新功能场景几乎：
- 不质疑必要性（PM 说要就要）
- 不搜索复用可能（直接从零开始设计）
- 不标注影响范围（新增功能对其他模块的涟漪效应未被预判）

这三个缺口分别对应三个思维维度：合理性检查、复用性扫描、影响范围分析。

---

## 2. 三个思维维度

### 2.1 合理性检查

**现状**：Agent 对 PM 描述的需求照单全收，不会主动质疑"这个需求是否真实"。

**思路**：在"深度探询"小节末尾增加触发条件——当 `asIs` 为空时自动切换至新功能评估模式。新增 `### 新功能评估` 小节（位于"深度探询"之后），`一、合理性自检` 作为该小节的第一部分。

**关键问题**：
| 问题 | 意图 |
|------|------|
| 这个需求是真实缺口，还是已有功能没被 PM 找到？ | 防止重复建设 |
| 如果不做，业务影响有多大？用户当前绕过成本是多少？ | 判断需求紧急度 |
| 有没有比 PM 描述的更简单方案？配置化/加字段/改参数能否解决？ | 降低实现成本 |
| MVP 边界在哪？哪些场景可以二期再做、哪些可以先人工过渡？ | 控制首次交付范围 |
| 这个需求的 ROI 如何？以对系统的了解，评估开发成本与业务价值比 | 辅助优先级判断 |

**Prompt 文本**（提议）：

```
**合理性自检**（自问自答，非追问PM）
1. 真实缺口还是已有功能没被找到？grep菜单/按钮关键词
2. 不做的业务影响？用户当前绕过成本？
3. 有更简单替代方案？配置化/加字段能否解决？
4. MVP边界？哪些场景二期做或人工过渡？
```

**备注**：第四问"MVP 边界"对处于快速迭代期的产品尤为重要——Agent 应当主动识别"什么可以做进去、什么可以留到二期"，而不是等 PM 来拆分。

---

### 2.2 复用性检查

**现状**：Agent 面对全新需求时，倾向于从零设计方案，而非扫描现有代码中可复用的页面/组件/接口。

**思路**：在 `pm-prd.ts` 新增的"新功能评估"小节中，`二、复用性扫描` 作为第二部分。同时在 `pm-prd.ts` 现有"技术可行性线索"段补充：新功能场景下不读码验证现有行为，改为 grep 扫描可复用的现有页面/组件/接口。

**搜索模式**：

| 层面 | 搜索方法 | 示例 |
|------|----------|------|
| 前端页面 | grep 功能关键词匹配 vue 页面标题/路由路径 | `find workFront -name '*.vue' \| xargs grep -l '关键词'` |
| 后端 Controller | grep 模块名+资源名匹配 RequestMapping | `grep -r 'RequestMapping.*资源名' workBack/*/src/main/java` |
| 组件层面 | grep 既有 UI 组件模式 | el-table/el-form/el-dialog 在现有类似功能页面中的使用 |
| 跨模块 | grep 其他模块类似 Service 命名 | `grep -r '类似关键词' workBack/smsbm workBack/smbe workBack/smip` |
| 基础设施 | 检查现有报表引擎、审批流配置、附件组件、银行适配器通道 | 按模块特征扫描 |

**Prompt 文本**（提议）：

```
**复用性扫描**
1. 前端：grep功能词找最近似页面直接改编
2. 后端：grep模块+资源名找最近似Controller/Service
3. 跨模块：其他模块有类似逻辑？可抽象公共组件？
4. 基础设施：可复用报表引擎/审批流/银行适配器？
```

**备注**：复用性扫描的目的不是"找到一模一样的代码然后 copy"，而是"找到最近似的东西作为起点"。这对 Agent 的 grep 能力和模块理解提出了要求。跨模块扫描尤其重要——不同业务线（smsbm/smbe/smip）经常有高度相似的查询/审批/导出逻辑，但 Agent 往往只盯着当前模块 grep。

---

### 2.3 影响范围分析

**现状**：Agent 设计新功能时几乎不考虑对其他模块的影响——新增字段不查报表是否引用、新增页面不查权限是否需要新建、新增接口不查跨模块配置是否需调整。

**思路**：在 `pm-prd.ts` 新增的"新功能评估"小节中，`三、影响范围核查` 作为第三部分。同时修改 `pm-prd.ts` "最小公约"中的 `readyToFinalize` 条件，增加新功能豁免子句。并在"深度探询"的 `openQuestions` 分类中增加 `[MVP]` 和 `[ROI]` 两个标注类别。

**影响维度**：

| 维度 | 关键问题 |
|------|----------|
| **数据流影响** | 新增字段/表是否影响现有报表、统计图表、导出 Excel 功能？是否需同步修改关联模块的数据查询 SQL？ |
| **权限安全** | 新功能/新页面是否需要新增权限点？现有菜单角色是否需要重新赋权？数据按什么范围隔离（省份/区县/机构）？ |
| **跨模块配置** | smip 模块的 `provinceOverride` 是否需新增省份差异化配置？银行适配器是否需新增渠道或修改协议参数？ |
| **性能容量** | 批量操作有无上限、分页、超时设计？数据量级预估？现有 Redis/DB 能否支撑？ |
| **兼容回滚** | 旧数据/旧流程是否需向后兼容？上线出问题的回滚方案是什么？是否需要灰度开关控制？ |
| **安全审计** | 新增接口需要登录鉴权吗？防越权校验？敏感字段脱敏？关键操作是否记审计日志？ |

**Prompt 文本**（提议）：

```
**影响范围核查**
1. 数据流：新增字段影响报表/统计/导出？
2. 权限：需新权限点？现有角色需重新赋权？
3. 跨模块：smip省份配置/银行适配需调整？
4. 性能：批量上限分页超时考虑？
5. 兼容：旧数据向后兼容？回滚方案？
6. 安全：接口鉴权防越权？敏感字段脱敏？
```

**备注**：影响范围分析是三个维度中**最容易工具化**的。如果高频影响路径被验证为稳定，可以产出 `hints` 文件覆盖常见影响路径（如"新增字段时检查哪些报表""新功能上线时走什么灰度流程"）。

## 3. 具体 Prompt 改动建议

### 在 `pm-prd.ts` 中新增"新功能评估"小节

插入位置：在 `### 最小公约` 与 `### 深度探询` 之间（或 `### 深度探询` 之后），独立成段。

**新增小节内容**（~487 chars）：

```
### 新功能评估（全新需求场景）

当PM需求在代码中找不到已有实现（asIs为空）时，进入新功能模式：

**合理性自检**（自问自答）
1. 真实缺口还是已有功能没被找到？grep菜单/按钮关键词
2. 不做的业务影响？用户当前绕过成本？
3. 有更简单替代方案？配置化/加字段能否解决？
4. MVP边界？哪些场景二期做或人工过渡？

**复用性扫描**
1. 前端：grep功能词找最近似页面直接改编
2. 后端：grep模块+资源名找最近似Controller/Service
3. 跨模块：其他模块有类似逻辑？可抽象公共组件？
4. 基础设施：可复用报表引擎/审批流/银行适配器？

**影响范围核查**
1. 数据流：新增字段影响报表/统计/导出？
2. 权限：需新权限点？现有角色需重新赋权？
3. 跨模块：smip省份配置/银行适配需调整？
4. 性能：批量上限分页超时考虑？
5. 兼容：旧数据向后兼容？回滚方案？
6. 安全：接口鉴权防越权？敏感字段脱敏？

**readyToFinalize豁免**：新功能asIs空属正常，标注[MVP方案简述][影响范围结论]即可finalize，无需asIs/codePaths有内容。

openQuestions新增[MVP]和[ROI]标注：前者记二期范围，后者记该功能的价值判断
```

### 现有 `pm-prd.ts` 对应修改点

| 位置 | 改动 |
|------|------|
| "最小公约" → `readyToFinalize` 条件 | 增加新功能豁免子句：asIs 为空且已标注 `[MVP方案简述]` + `[影响范围结论]` 可 finalize |
| "深度探询" → openQuestions 分类 | 新增 `[MVP]`（二期范围）和 `[ROI]`（价值判断）两个标注类别 |
| "技术可行性线索" | 补充：新功能场景下不读码验证现有行为，改为 grep 扫描可复用的现有页面/组件/接口 |
| "深度探询" → 末尾 | 增加触发条件：当 asIs 为空时自动切换至新功能评估模式 |

## 4. 测试场景设计

### 场景一：新增基金预算执行监控大屏

**PM 描述**：需要一个基金预算执行监控大屏，实时展示全省基金收支进度，各险种分开展示，最好有趋势图。

**Old 行为（无新功能模式）**：
1. 创建空的 `asIs`（找不到现有代码）
2. grep "大屏""监控"无果后困惑
3. 反复追问 PM "数据源？刷新频率？图表组件？"
4. `codePaths` 无法填 -> 违反 `readyToFinalize` 条件 -> 卡死
5. `openQuestions` 堆积技术细节，PM 无法回答导致死锁

**New 行为（有新功能模式）**：

| 阶段 | Agent 行为 |
|------|-----------|
| 1. 识别模式 | `asIs` 为空 -> 识别为全新功能 -> 进入新功能评估模式 |
| 2. 合理性自检 | 真实需求，grep 确认无类似页面；用户当前靠手工查报表做监控，成本高 |
| 3. 复用性扫描 | 发现现有报表页有 ECharts 趋势图组件可直接复用；发现现有基金结余查询接口可做数据源 |
| 4. 影响范围 | 数据口径需对齐现有基金报表；需新增菜单权限点；首次加载建议限最近 12 个月 |
| 5. MVP 边界 | 先做日报+固定指标（覆盖 80%），二期做实时推送+拖拽配置 |
| 6. openQuestions 标注 | `[MVP:日报先行]` `[ROI:中等]`——替代了原来的"图表组件用什么？"等技术追问 |
| 7. readyToFinalize | 标注 `[MVP方案简述]` `[影响范围自检通过]`，无需 `asIs`/`codePaths` 有内容 |

---

### 场景二：支付计划列表批量驳回

**PM 描述**：支付计划列表加一个批量驳回按钮，勾选多条后一键驳回，驳回原因统一填写。

**Old 行为（无新功能模式）**：
1. grep 找到单条驳回 API
2. `asIs` 写"现在只能逐条驳回"
3. `codePaths` 直接引用驳回接口
4. `toBe` 照 PM 描述写批量逻辑
5. `readyToFinalize` 时发现批量场景边界条件（部分失败/审计日志/权限校验）未验证 -> deadlock

**New 行为（有新功能模式）**：

| 阶段 | Agent 行为 |
|------|-----------|
| 1. 识别模式 | `asIs` 有"现有逐条驳回接口" -> 不是全新功能 -> 仅触发影响范围核查 |
| 2. 复用性扫描 | grep 现有驳回接口（smbe/PaymentController）可直接扩展为批量 |
| 3. 影响范围 | 批量需校验多条权限；审计日志需批量记录；部分失败策略需确认 |
| 4. MVP 边界 | 先批量驳回+逐条校验（一条失败不拦其他），二期加批量审批 |
| 5. openQuestions 标注 | `[边界:部分失败策略]` `[审计:每笔单独记日志?]` |

**注意**：这个场景是"功能增强"而非"全新功能"，所以只触发影响范围核查，不触发完全的合理性自检和复用性扫描。这说明三个维度不是绑定使用的——应根据 `asIs` 是否为空来分级触发。

### 场景三：跨省基金调度

**PM 描述**：省级账户之间可以互相划转基金余额，A 省多余额度调给 B 省，资金从 A 省归集户转到 B 省支出户。

**Old 行为（无新功能模式）**：
1. grep smbe/smsbm/smip 等所有模块找不到功能 -> `asIs` 空
2. 陷入读码困境：读银行接口、转账 Controller、账户余额
3. `codePaths` 写了大量银行接口类名（违反 PM 可读要求）
4. `readyToFinalize` 卡死：`asIs`/`codePaths` 无法填
5. 退化成追问 PM "银行接口支持跨省？"等研发细节

**New 行为（有新功能模式）**：

| 阶段 | Agent 行为 |
|------|-----------|
| 1. 识别模式 | `asIs` 空 -> 识别为全新跨模块功能 -> 新功能评估模式 |
| 2. 合理性自检 | 真实缺口；当前线下开票划转流程繁琐；高 ROI |
| 3. 复用性扫描 | smbe 支付接口做底层通道；银行适配器有银企直连；基金账户模块有余额 Service |
| 4. 影响范围 | 两省余额变动影响月报统计；需省级财政审批+双人复核；smip 需配超收比例；银行需确认跨省转账支持 |
| 5. MVP 边界 | 先省对省单笔调度（线下审批+系统执行），二期自动调拨 |
| 6. openQuestions 标注 | `[权限:审批流程待确认]` `[银行:跨省限额待确认]` `[MVP:二期自动调拨]` |

**注意**：这个场景暴露了当前 prompt 的一个深层问题——`codePaths` 的设计假定"每条需求都对应一段现有代码"。对于跨省基金调度这种跨模块、跨系统的需求，`codePaths` 应该写什么？如果写银行接口类名，违反"PM 读者"可读要求；如果不写，`readyToFinalize` 过不了。新功能模式的 `readyToFinalize` 豁免正是为了解决这个矛盾。

## 5. 开放讨论点

### 5.1 三个维度是新增章节还是融入"深度探询"？

**两种方案**：

| 方案 | 优点 | 风险 |
|------|------|------|
| **A: 独立 "新功能评估" 章节** | 结构清晰，Agent 在 asIs 为空时明确知道切换模式 | 新增约 500 chars，接近 trimPmRules 上限 |
| **B: 融入 "深度探询" 小节** | 不新增章节，保持 prompt 结构稳定 | 新功能场景的思维模式与"深度探询"（追问 PM）混在一起，Agent 可能混淆"自问自答"和"追问 PM" |

**倾向**：方案 A。三个维度的思维模式与"深度探询"有本质区别——深度探询是追问 PM，新功能评估是 Agent 自问自答。混在一起会导致 Agent 把合理性自检的问题拿去问 PM。

### 5.2 Prompt 长度控制：新章节是否触发 trimPmRules 的 2800 限制？

当前 `PM_PRD_RULES` 约 2600 chars。新增"新功能评估"小节约 487 chars，总计约 3100 chars，超出 2800 的限制。需要考虑：

- 增大 `trimPmRules` 上限（推荐 3200）
- 或精简现有 prompt 中冗余内容腾出空间
- 或仅在 `asIs` 为空时动态追加该段（而非作为固定 prompt）

**推荐**：动态追加。在 `format-turn-prefix.ts` 的 prd mode hint 中，当 `asIs` 为空时追加新功能评估指引。这样既不影响现有流程，也不增加固定 prompt 长度。

### 5.3 合理性质疑会不会让 Agent 变得"抬杠"？怎么平衡？

这是最大的 UX 风险。必须明确区分：

- **做**：Agent 内心自问自答"这个需求是否真实""是否有更简单方案"——结果写在 `openQuestions` 或 `toBe` 的"待确认"标注中
- **不做**：Agent 直接反问 PM"你这个需求真的有必要吗？""有没有更简单的方法？"

**边界**：除非 PM 提出了明显可识别的"重复需求"（如 PM 说要做一个功能，Agent grep 发现已存在），否则合理性自检的结论不应以反问的形式抛给 PM。它应当影响的是 Agent 对 `toBe` 的表述方式（如标注 MVP 范围），而非对话的语气。

**自检规则**：合理性自检的输出进入 `openQuestions[MVP]` 和 `openQuestions[ROI]` 标注，以及 `toBe` 中的 MVP 范围说明。不直接写入 `blockingQuestion`。

### 5.4 复用性搜索的具体 grep 模式怎么写才能高效？

复用性搜索的关键是"翻译"——将 PM 描述的业务语言转换为代码搜索的技术关键词。这需要：

1. **关键词提取**：从 PM 描述中提取核心业务名词（如"基金预算执行监控" → "基金""预算""监控""大屏"）
2. **分层 grep**：先前端页面（.vue），再后端 Controller，再跨模块 Service
3. **模式匹配**：不只是关键词，还要匹配命名模式（如 `*Controller.java` 中的 `@RequestMapping` 路径）

**建议的 grep 模式**：

```bash
# 前端页面：匹配路由 path 和页面 title
grep -r 'path:\|title:' workFront/src/views/ | grep -i '关键词'

# 后端 Controller：匹配 RequestMapping 路径
grep -r 'RequestMapping.*' workBack/ | grep -i '资源名'

# 跨模块 Service：按能力扫描
for m in smsbm smbe smip smr; do
  grep -r 'class.*Service' workBack/$m/src/main/java | grep -i '关键词'
done
```

**落地方式**：这些 grep 模式可以固化到 Agent 的工具使用指引中，或者在 `technical-feasibility` 提示中增加"新功能场景下优先按以下模式 grep……"。

### 5.5 影响范围分析是否需要配合工具（如新增 analyze_impact 工具）？

当前依赖 Agent 的 grep 能力和模块理解。如果影响范围分析的高频路径被验证为稳定，可以考虑工具化：

- **轻量方案**：产出 `hints` 文件覆盖常见影响路径（如"新增字段时检查哪些报表""新功能上线时走什么灰度流程"）
- **重量方案**：新增 `analyze_impact` 工具，接受"新增模块/字段/页面"参数，自动 grep 并汇总影响

**推荐**：先轻量方案。运行一段时间积累高频影响路径后，再评估是否需要专用工具。

### 5.6 是否需要更新 hints 文件来覆盖"常见影响路径"？

需要。hints 文件目前主要用于业务上下文理解（模块分工、银行适配器等）。可以扩展一个 `hints/impact-paths/` 目录，按模块存放"常见影响清单"，如：

- `smbe/payment-impacts.md`：修改支付相关功能时需检查的报表、统计、权限点
- `smip/province-config-impacts.md`：新增省份配置时需调整的模块清单
- `smsbm/fund-report-impacts.md`：新增基金字段时需同步修改的报表清单

这样 Agent 在影响范围核查阶段可以直接 read hints 文件，而不是完全依赖 grep。

## 附录：与现有 prompt 架构的兼容性

### 与 MEMORY_GUIDANCE 的关系

新功能评估模式与跨会话记忆（M0）没有直接冲突。合理性自检的结果不应写入 memory（遵循"任务进度不存记忆"原则），但"常见影响路径"的发现可以作为 hints 持久化。

### 与 "最小公约" 的关系

新功能模式的 `readyToFinalize` 豁免是对"最小公约"出口条件的补充，而非替代。在 `asIs` 有内容的情况下，出口条件不变。仅在 `asIs` 为空且已标注 `[MVP方案简述]` + `[影响范围结论]` 时启用豁免。

### 与 "深度探询" 的关系

新功能评估的"自问自答"模式与"深度探询"的"追问 PM"模式共存但不重叠。触发条件区分：

| 条件 | 模式 |
|------|------|
| `asIs` 有内容（改现有逻辑） | "深度探询"追问 PM + "技术可行性线索"读码验证 |
| `asIs` 为空（新功能） | "深度探询"追问 PM + "新功能评估"自问自答 |
| 混合场景（部分有、部分无） | 两者组合使用，已有部分走深度探询，新部分走新功能评估 |

### 与 turn-prefix 的关系

推荐在 `format-turn-prefix.ts` 的 prd mode hint 中动态追加新功能评估指引——当进入 prd 模式且 `asIs` 为空时，mode_hint 中增加"当前识别为全新功能场景，请按新功能评估模式执行"。这样不动固定 prompt，按需加载。
