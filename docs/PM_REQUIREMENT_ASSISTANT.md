# PM 需求助手 — 需求背景与方案说明

| 项目 | 内容 |
|------|------|
| 产品 | letsTalk — 面向代码库的 Agent 对话（Vue + Java monorepo） |
| 主要用户 | **产品经理** |
| 状态 | **v0.2** — 已吸收外部架构评审，待实现 |
| 日期 | 2026-05-27 |

---

## 1. 背景与问题（不变）

- PM 输入常混乱、可混合多条；PM **不应**装懂全站业务逻辑与影响面。
- 研发第一性原理：**修改** = 哪页哪按钮 + 改什么逻辑；**新增** = 表格 + 配置表 + 多操作（CRUD/工作流/业务动作）。
- letsTalk 价值：**以代码为准** + 帮 PM 收成研发可评审的说明（缺项标出，仍出完整稿）。

---

## 2. 对 v0.1 评审的结论（是否采纳）

| 评审点 |  verdict | 说明 |
|--------|--------|------|
| 纯对话维护隐形表单 = 审问/挤牙膏 | **采纳** | 改为 **对话 + 右侧需求草稿板（双轨）** |
| 定稿前「全局扫描」 | **采纳修正** | 删除「全局扫描」；改为 **基于已确认代码锚点的受击面分析** |
| 软触发定稿 | **采纳** | 昂贵动作必须 **PM 点击按钮**（Human-in-the-loop） |
| 无锚点则无法核对代码 | **采纳** | **无锚点、不核对代码**；优先用 hints/菜单名逼出锚点 |
| 每轮 2～4 问 | **采纳修正** | 对话里 **最多 1 个阻断性问题**；其余缺口在右侧 **标红** |
| 拆条可合并 | **采纳** | 对话或右侧板 **合并/拆分** |
| 两套 PRD 模板 | **采纳** | **一套模板**，无字段章节标「无」或折叠 |
| Jira 对接 | **采纳** | M1 仅 **Markdown 导出/复制** |
| 下线「写需求 = 整段 PRD 注入」 | **采纳** | 模板内容改为 System Prompt **核对清单** + 右侧结构化状态 |

**保留的 nuance（评审未写清）：**

- 受击面 **预算**：单次定稿分析控制在约 **2～3 次工具调用 / 30 秒内**（grep 引用链、read_method），避免 Token 与等待失控。
- M1 右侧板可先 **只读展示** Agent 维护的状态，M1.5 再支持 PM **点选填空/合并**。
- 「最多 1 个问题」指 **阻断开工** 的项；非阻断缺口 **只更新右侧，不刷屏提问**。

## 3. 目标形态（v0.2）

### 3.1 双轨视图（Dual-Track）

```text
┌─────────────────────┬──────────────────────────────┐
│  对话流（左）          │  需求草稿板（右）                │
│  PM 自由描述          │  需求① 导出  🟡 缺验收标准        │
│  Agent 复述/警告      │  需求② 重置确认 🟢 已就绪         │
│  最多 1 个关键追问     │  PM 可点击字段补全 / 合并条目     │
└─────────────────────┴──────────────────────────────┘
```

- Agent 每轮在后台更新结构化状态（建议：`tool` / SSE 事件 `requirement_state` → JSON）。
- 对话语气：**「已在右侧拆成两条；导出还缺导出列范围，方便补一下吗？」** 而非连珠炮问卷。

### 3.2 锚点破冰漏斗

| 条件 | 行为 |
|------|------|
| 无锚点 | **不** read/grep 核对；Agent 用 `.agent/hints/`、锚点列表、菜单名匹配，帮 PM 锁定页面 |
| 有锚点 | 可核对现状；填充草稿板「现状」「关联 path」 |
| 有锚点 + 已落到具体改动点（如某 Controller 方法） | 才允许 **受击面分析** |

### 3.3 受击面分析（Blast Radius，非全局扫描）

**禁止**：未落代码点前做「全库逻辑扫描」。

**允许**（定稿按钮点击后，针对已确认锚点）：

- **上游**：谁调用了这个接口/方法/组件；
- **下游**：该方法引用了哪些公共 Config/工具类（含 yml 读取链，规则可后续写入 `.agent/memory`）。

输出用语：**可能受击** / **建议研发确认**，分「共用底座 / 分省或范围待确认」。

### 3.4 定稿：仅按钮触发

Agent 认为可定稿时，在对话中展示 **可操作 UI**（Generative UI / 服务端下发 action）：

- `[ 扫描受击面并生成开发说明 ]` — 触发工具链 + 最终 Markdown；
- 可选 `[ 仅生成文档（跳过受击面）]` — 省时、风险自负，需二次确认。

**禁止**仅凭 PM 口语「整理一下吧」自动进入重型分析（可说「准备好了请点右侧按钮」）。

### 3.5 输出物

- **一套**开发说明模板（修改/新增由 Agent 填「无」节）。
- 固定章节：**已确认 / 待确认 / 受击面提示 / 验收 / 开放问题**。
- 缺项 **不阻塞生成**，但必须 **【待确认】** 可见。

## 4. 自动拆条规则

- 默认从 PM 第一段有效描述 **自动拆条**。
- PM 可说「不要拆」或右侧 **合并**；Agent 服从。
- 每条需求状态：`draft` | `ready` | `blocked` | `conflict`（与代码不一致）。

## 5. 与现有实现的关系

| 现有 | v0.2 处理 |
|------|-----------|
| 「探索 / 写需求」模式切换 | **下线写需求 JIT 大段模板**；探索保留；需求整理 = 进入 PM 助手流程（双轨+草稿板） |
| 锚点侧栏 | 保留；升为 **硬性前置**（无锚点则破冰） |
| grep / read / list_methods | M2 核对；M3 受击面 |
| 导出 Markdown | M3 定稿后 |
| `packages/context` 的 `pm_rules` / `prd_template` 每轮注入 | **改为** 清单在 Prompt + 状态在右侧 JSON，避免每轮重复灌模板全文 |

## 6. 分阶段 Milestone

### M1 — 双轨视图与拆条（The Dual-Track Canvas）

- 前端：左 Chat + 右 **只读** 草稿板（条目、字段、🟢🟡🔴）。
- Agent：`update_requirement_state`（或等价 SSE）— 听懂口述 → 拆条 → 更新 JSON。
- 有锚点时：可读 anchor 预览 / 简述现状；**不做**受击面。
- 下线旧「写需求」模板注入。

### M2 — 代码现实核对（The Reality Check）

- 条目绑定代码锚点（vue / Controller 方法）后：`read_method` / `grep` 核对 PM 陈述。
- 冲突 → 对话警告 + 草稿板 `conflict` + 文档【待业务确认】。

### M3 — 受击面与定稿（Blast Radius & Finalize）

- 定稿按钮 → 受击面（2～3 次 tool，锚定入口）→ 生成完整 Markdown → 导出。

## 7. 示例（v0.2 交互）

**PM：** Detail 页重置要加确认，列表要导出。（未选锚点）

**Agent：** 我先帮你对齐页面：是指【报销明细 Detail】吗？（匹配 hints / 锚点列表）请确认或左侧选锚点。

**PM：** 对，就 Detail。

**Agent：** 已在右侧拆成 2 条。重置那条我对过代码，目前有 X 逻辑。【对话里 1 问】确认后是否允许取消？导出条还缺「导出哪些列」，可在右侧补。

**PM：** （在右侧填列名）… 好了。

**Agent：** 显示按钮 `[ 扫描受击面并生成开发说明 ]`。

**PM：** *点击*

**Agent：** （对 `DetailController.reset` 等做引用 grep）→ 输出 Markdown，含受击面与待确认项。
