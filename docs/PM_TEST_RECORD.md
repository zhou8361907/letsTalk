# PM 场景测试记录 — 基金支出支付计划

## 测试方法

以 PM 口吻转述现场人员的自然语言需求，逐轮补充信息，观察 Agent 在 PRD 模式下的行为。
目的是发现 prompt 和系统的问题，而非验证功能。

**当前验收标准（最小公约 + Phase 0 自检，2026-06-01 起）**

- 允许首轮就有 1 条 draft，但 toBe **不得**写 PM 未说的规则（应「待确认：…」或 pending）
- 须有额度口径相关的 blockingQuestion 或 openQuestions
- 对话正文无类名/方法名；读码线索进 codePaths
- readyToFinalize 仅在公约齐时生效（无 blocking、toBe/acceptance 满且无待确认）

## 第 1 轮测试（修复前 · 2026-05）

### 用户输入

> 基金支出那边的同事反馈，现在每个月做支付计划效率很低。流程是先汇总各县区的申请，再手动算可用额度，最后生成支付计划。但额度经常算错，而且支付计划生成后要一个一个点提交审批，很费时间。希望能自动算可用额度，支付计划做好后能批量提交审批。

### Agent 行为观察（修复前）

| 观察项 | 表现 | 评价 |
|--------|------|------|
| 是否探索代码 | ✅ grep 找到了 `PayPlanController` 和相关页面 | 好 |
| 是否追问模糊信息 | ❌ 没有追问"可用额度"的计算口径 | 差 |
| 第一轮是否写清单 | ❌ 直接产出 2 条结构化需求 | 差 |
| 技术细节是否暴露给 PM | ❌ 对话正文出现 `submit()` 等 | 差 |
| 需求条目语言 | ✅ 业务语言 | 好 |

### 根因（已处理）

- `PM_PRD_RULES` 缺最小公约与 update 前 toBe 自检
- 缺 readyToFinalize / 公约缺口与 store 联动

## 第 1 轮复测（修复后 · 待跑）

### 期望

| 观察项 | 期望 |
|--------|------|
| 清单 | 0～1 条；toBe 含 PM 原意 +「口径待确认」类表述，非写死公式 |
| 追问 | blocking 或 open 含「可用额度按什么算」 |
| 对话 | 1～2 句现状（业务话）+ 至多 1 个关键问 |
| 技术 | codePaths 有线索；正文无 submit/Controller |

### 待验证项

- [ ] 首轮 toBe 自检（无猜规则定稿）
- [ ] 额度口径 blocking/open
- [ ] 补充信息时更新同 id 而非新建第二条
- [ ] readyToFinalize 在有待确认时不亮
- [ ] 探索→需求整理切换后复用 asIs
