# AI Requirement OS — V1 页面分析规范

这份文档用于约束当前 V1 阶段的核心输入输出。  
V1 的新定位不再是“秒级草稿”，而是 **AI Requirement OS 的全局导航目录（Business Map）**：

- 允许花费更长时间做跨前后端侦察
- 优先产出高置信度的业务落点和代码坐标
- 为后续沙箱 Agent 提供可靠的“黄页”

---

## 1. V1 目标

V1 只做一件事：

**给定一个页面，生成一份产品可读、研发可用、且可供 Agent 继续深入读取代码的业务地图。**

当前主链路：

```text
页面路径 + 前后端路径
    ↓
页面工作区组装
    ↓
Evidence Bundle（方法级裁剪）
    ↓
Agent 直接输出 PageDataLineage JSON
    ↓
Markdown Renderer
    ↓
页面数据解析报告
```

这里有两个非常重要的原则：

1. **JSON 是主产物**
2. **Markdown 只是 JSON 的展示层**

---

## 2. 输入边界

V1 的输入尽量少，只保留当前真正必要的信息。

### 2.1 必要输入

```json
{
  "project_name": "",
  "frontend_path": "",
  "backend_path": "",
  "page_path": ""
}
```

说明：

- `project_name`：当前项目标识
- `frontend_path`：前端工程根路径
- `backend_path`：后端工程根路径
- `page_path`：当前要分析的具体 Vue 页面

### 2.2 Agent 可使用的上下文

Agent 在生成结果时，允许读取：

- 当前页面 `.vue`
- 当前页面直接引用的前端 API 文件
- 能直接追踪到的后端 `Controller`
- 必要时继续追踪到 `Service`、`DTO`、`VO`、`Mapper`

当前阶段不要求 V1 穷尽所有数据库血缘，也不要求一次性跑完整个企业系统。
但要求它尽量做到：

- 前端 API -> Controller 方法
- Controller 方法 -> Service 方法
- 给出精准绝对路径与方法入口
- 遇到复杂逻辑时明确承认未知，而不是硬编总结

补充原则：

- 那些可以通过后续工具调用快速再次查到的细碎实现信息，不必在 V1 主报告里堆满
- V1 报告负责沉淀高价值结论
- 更细的代码定位、字段拆解、范围条件细节，可在 V2 问答阶段由 Agent 二次搜索代码补充

## 4. 字段解释与取舍

### 4.1 `page_info`

用于说明当前页面是谁、在哪、主要做什么。

- `page_name`：页面名称
- `page_path`：页面文件路径
- `module_name`：所属模块
- `summary`：一句话说明页面职责

### 4.2 `search_fields`

表示页面查询区域的字段。

当前目标：

- 尽量列全
- 不必把字段说明写得特别深
- 能关联到后端字段最好，关联不到也可以先留空
- 像时间范围、区间条件这类容易从代码再次定位的实现细节，不强求在 V1 中做过细拆解

### 4.3 `display_fields`

表示页面展示区字段，当前阶段默认只关注**主表格字段**。

当前目标：

- 优先覆盖产品实际看到的展示字段
- 当前先只覆盖主表格列，不展开弹窗表单和详情区字段
- 后端字段只做轻量映射
- 不强求一开始追到数据库列

### 4.4 `initial_fetches`

表示页面初始化时的请求链路，必须是**有顺序的数组**。

当前重点：

- 请求接口
- 请求方法
- 前端入口方法
- 后端入口方法
- 作用概述
- 返回概述

### 4.5 `actions`

表示按钮或关键交互行为。

当前阶段最重要的是：

- 前端触发点
- 调用哪些接口
- 按什么顺序调用
- 传了哪些关键参数
- 大概做了什么
- 返回什么
- 后端精确落在哪里

收录原则：

- **优先收录会触发接口、影响业务数据或状态流转的关键操作**
- 如果只是打开弹窗、切换页签、重置查询这类过于简单的前端行为，当前阶段可以不写进主报告
- 如果一个操作链路很简单，就用简短条目写清，不需要为了“完整”而堆太多过程噪音
- 如果一个操作涉及多次关键接口调用，则保留清晰顺序，便于后续 Agent 继续追踪

这里不要求把每个按钮都写成超深的后端审计报告，但至少要做到：

- 能顺着 `request_flow -> backend_pointers` 找到真实代码
- Controller / Service 入口必须给绝对路径
- 若后端逻辑太深，则必须明确停在什么地方

### 4.6 `backend_pointers`

这是 V1 新增的关键坐标字段，专门服务后续 Agent。

每个 `request_flow` 节点都应尽量补出：

- `layer`：`controller` / `service`
- `class_name`
- `method_name`
- `method_signature`
- `file_path`：绝对路径
- `line_number_hint`
- `note`

后续 Agent 如果要继续读代码，应优先使用这里的坐标，而不是重新盲搜全项目。
其中：

- `class_name + method_name + method_signature` 是稳定定位锚点
- `line_number_hint` 只做人类阅读提示，不应作为 Tool 定位主键

### 4.7 `backend_overview`

只是页面后端落点的简表，不求太全。

- `controllers`
- `services`
- `tables`

如果表没找到，可以为空。

### 4.8 `notes`、`unknowns` 与 `deep_logic_flags`

- `notes`：补充说明
- `unknowns`：当前还不能稳定确认的部分
- `deep_logic_flags`：当前已经定位到后端入口，但判断其逻辑过深，不适合在 V1 强行总结的部分

如果某个字段、接口返回、后端映射不明确，宁可放到 `unknowns`，也不要编造。

如果某个 Controller / Service 方法已经找到，但出现以下情况，也应该主动打标：

- 正则或轻量解析失败
- Service 级联调用过深
- 复杂权限、事务、模板或文件流逻辑很多
- 继续总结的幻觉风险明显变高

示例：

```json
{
  "deep_logic_flags": {
    "action:updateDetail:update": "update 的后端逻辑较复杂（涉及较多条件分支），V1 未继续深入总结"
  }
}
```

展示约定：

- `unknowns` 默认保留在结构化 JSON 中
- Markdown 报告 v1 默认不展示 `unknowns`
- 后续如果进入研发排查模式，再考虑把 `unknowns` 单独展开

## 5. Markdown 报告模板 v1

Markdown 报告面向产品、实施、研发协同阅读，所以要求：

- 可快速浏览
- 少堆实现细节
- 保留接口与字段主线
- 不展示过多源码证据

推荐模板如下：

```markdown
# 页面数据解析报告：[页面名称]

## 1. 页面概览
- 页面路径：`...`
- 所属模块：...
- 页面说明：...

## 2. 查询条件字段
| 页面展示名称 | 前端字段 | 后端字段 | 说明 |
| :--- | :--- | :--- | :--- |
| ... | `...` | `...` | ... |

## 3. 展示字段
| 页面展示名称 | 前端字段 | 后端字段 | 说明 |
| :--- | :--- | :--- | :--- |
| ... | `...` | `...` | ... |

## 4. 页面初始化请求
### 1. [请求名称]
- 请求接口：`GET /xxx`
- 前端入口：...
- 后端入口：...
- 接口作用：...
- 返回内容：...

## 5. 页面操作
### [操作名称]
- 前端方法：...
- 操作说明：...
- 请求链路：
  - 1. `POST /xxx`
    - 参数：id / status
    - 逻辑：...
    - 返回：...

## 6. 后端概览
- Controller：...
- Service：...
- 涉及数据表：...

## 7. 备注
- ...
```

说明：

- Markdown 报告 v1 默认不展示 `unknowns`
- 待确认项仍保留在 JSON 中，供后续 Agent 问答、研发排查或二次分析使用

## 6. Agent Prompt v1 设计原则

当前阶段，Prompt 不追求花哨，而要追求稳定。

### 6.1 Agent 角色

Agent 当前应该被定义成：

**一个精通 Vue、Spring Boot、MyBatis 的页面数据解析助手。**

它的职责不是改代码，而是：

- 理解页面结构
- 找出字段映射
- 追踪初始化接口
- 追踪关键按钮操作
- 输出结构化 JSON

同时要明确：

- 它不是一次性把所有代码细节塞进报告
- 它后续还可以继续调用工具，重复搜索代码、补充结论、回答问题
- 所以 V1 先沉淀稳定骨架，不追求把所有可搜索事实都塞进首版文档

### 6.2 Prompt 核心约束

Prompt 必须反复强调下面几点：

1. 只处理当前页面，不分析整个项目
2. 只能使用当前允许的页面级上下文
3. 不要编造字段、接口、按钮、数据库表
4. 找不到就留空或写入 `unknowns`
5. 重点看字段、接口、参数、返回和动作链路
6. 忽略样式、动画、布局细节
7. 能被后续工具搜索轻松补充的细枝末节，不必过度展开
8. 输出只能是结构化 `PageDataLineage`

### 6.3 当前推荐工作流

推荐在 Prompt 里隐含下面这条分析顺序：

1. 先看 `.vue` 页面
2. 找搜索区字段和展示字段
3. 找初始化请求
4. 找按钮行为和前端请求链路
5. 顺着接口追到后端入口
6. 补充轻量的后端概览
7. 输出 JSON

---

## 7. V1 成功标准

当下面几件事稳定成立时，就可以认为 V1 基本跑通：

1. 选择一个页面后，能稳定生成 `PageDataLineage`
2. 同一个页面重复生成时，结果整体一致
3. JSON 中字段、初始化请求、主要操作能基本覆盖页面主流程
4. Markdown 报告能让产品快速看懂页面职责和接口行为
5. 遇到拿不准的地方，系统会明确标注 `unknowns`，而不是胡写

---

## 8. 当前不做什么

为了守住节奏，下面这些内容明确不属于 V1：

- 真实代码自动修改
- 复杂沙箱交互
- 深度数据库血缘审计
- 全项目图谱构建
- 多 Agent 编排优化
- 面向最终交付的高保真产品原型

---

## 9. 下一步

V1 规范定完后，接下来按这个顺序推进：

1. 让后端 Agent 真正生成 `PageDataLineage`
2. 用同一份 JSON 渲染 Markdown 报告
3. 把工作台里的占位结果替换成真实 Agent 输出
4. 再进入 V2 的页面问答 Agent
